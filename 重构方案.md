# Visual Vibe Coding Studio 重构方案

## 1. 当前判断

当前项目可以视为“高保真原型”，不是“按需求文档实现的可用系统”。

已有基础：

- 多工作区 UI 外壳
- Feature Tree 雏形
- Wireframe 画布雏形
- Tauri 工具调用基础
- AI 面板和上下文概念验证

核心缺口：

- 缺少项目创建与项目配置流程
- 缺少统一 Project Graph
- 产品到设计到开发到测试到运维的链路没有打通
- AI 仍以单体调用和模拟逻辑为主
- 开发工作区大量能力仍为静态数据或模拟执行

结论：

建议做“架构级重构”，保留技术栈和可复用 UI/工具基础，重建核心数据模型、流程编排和执行层。

## 2. 重构目标

重构后系统需要满足以下主线：

1. 创建项目并配置产品类型、技术栈、框架、UI 框架、部署方式
2. 输入需求并结构化为 Requirement / PRD / Feature Tree / Page Structure
3. 在画布中编辑 Wireframe，并保存为结构化 JSON
4. 由 Design Agent 基于 Wireframe 生成 UI Design / Design System
5. 由 Dev Agent 基于 Project Graph 生成前后端代码
6. 由 QA Agent 生成并执行测试
7. 由 DevOps Agent 生成部署配置并执行部署
8. 所有阶段共享统一 Project Graph 与 Project Memory

## 3. 重构原则

- 保留 `Tauri + React + TypeScript + Zustand` 技术栈
- 保留部分可复用的画布、组件库、Tauri 工具命令
- 先重建领域模型，再重建 UI
- 先打通主流程，再补体验优化
- AI 只读取相关节点，避免继续做“整页 prompt 拼接”
- 所有模块优先真实可执行，避免继续堆静态占位页面

## 4. 建议的新架构

### 4.1 分层

建议拆成五层：

1. Presentation
   - 项目创建页
   - 产品工作区
   - 设计工作区
   - 开发工作区
   - 测试工作区
   - 运维工作区

2. Application
   - 工作流编排
   - Agent 调度
   - 命令执行
   - 工具调用

3. Domain
   - Project Graph
   - Project Memory
   - Requirement / Feature / Page / Component / API / Test / Deploy 实体

4. Infrastructure
   - Tauri 文件系统
   - Shell 执行
   - AI Provider
   - 本地存储
   - JSON 持久化

5. AI Runtime
   - PM Agent
   - Design Agent
   - Dev Agent
   - QA Agent
   - DevOps Agent
   - Skills

### 4.2 核心数据中心

重构核心应从多个独立 store 改为“单一项目中心 + 模块视图 store”。

建议核心对象：

```ts
interface ProjectConfig {
  id: string;
  name: string;
  appType: 'web' | 'mobile' | 'mini_program' | 'desktop' | 'backend' | 'api';
  frontendFramework?: string;
  backendFramework?: string;
  database?: string;
  uiFramework?: string;
  deployment?: string;
}

interface GraphNodeBase {
  id: string;
  type: 'requirement' | 'prd' | 'feature' | 'page' | 'wireframe' | 'component' | 'api' | 'database' | 'test' | 'deploy';
  name: string;
  status?: 'draft' | 'ready' | 'in_progress' | 'done' | 'failed';
  metadata?: Record<string, unknown>;
}

interface GraphEdge {
  id: string;
  from: string;
  to: string;
  relation: 'derived_from' | 'contains' | 'implements' | 'tests' | 'deploys' | 'depends_on';
}

interface ProjectGraph {
  nodes: GraphNodeBase[];
  edges: GraphEdge[];
}

interface ProjectMemory {
  techStack: Record<string, string>;
  designSystem: Record<string, unknown>;
  codeStructure: Record<string, unknown>;
}
```

## 6. 模块重构建议

### 6.1 项目管理模块

需要新增：

- 创建项目向导
- 技术栈选择器
- 框架选择器
- UI 框架选择器
- 部署方式选择器
- 项目配置持久化

产出：

- `project.json`
- 初始化后的 `ProjectConfig`
- 初始 `ProjectMemory`

### 6.2 产品模块

需要新增：

- 原始需求录入
- PRD 生成与编辑
- Feature Tree 生成与编辑
- Page Structure 生成与编辑
- 节点之间的引用关系

产出：

- `requirements/*.json`
- `prd/*.json`
- `feature-tree.json`
- `page-structure.json`

### 6.3 设计模块

保留：

- 组件库
- 画布基础

需要重做：

- Page 级 wireframe 模型
- 组件属性面板绑定真实状态
- JSON 存储
- Feature/Page/Wireframe 关联
- Wireframe -> UI Design 任务

产出：

- `wireframes/*.json`
- `design-system.json`
- `ui-spec/*.json`

### 6.4 开发模块

需要重做：

- 真实文件树
- 真实代码编辑器
- 真实终端执行
- 基于 graph 的上下文注入
- 代码生成任务管理
- 生成结果回写 Artifact 节点

产出：

- 前端代码
- 后端代码
- API 定义
- 数据模型

### 6.5 测试模块

需要新增：

- 测试计划节点
- 单测、API 测试、E2E 测试生成
- 执行测试命令
- 测试报告展示
- Bug 修复触发链路

产出：

- `tests/unit`
- `tests/api`
- `tests/e2e`
- `reports/*.json`

### 6.6 运维模块

需要新增：

- 部署目标配置
- Dockerfile 生成
- CI 配置生成
- Deployment Script 生成
- 部署历史记录

产出：

- `Dockerfile`
- `.github/workflows/*.yml` 或其他 CI 配置
- `deploy/*.sh`

## 7. Agent 与 Skill 设计

### 7.1 Agent

建议固定 5 个 Agent：

- PM Agent
  - 需求解析
  - PRD 生成
  - Feature Tree 生成

- Design Agent
  - Wireframe 转 UI Design
  - 设计系统提取

- Dev Agent
  - 前后端代码生成
  - 增量修改

- QA Agent
  - 测试生成
  - 测试分析
  - Bug 修复建议

- DevOps Agent
  - Docker / CI / Deployment 生成

### 7.2 Skill

建议先做 7 个内置 Skill：

- `generate_prd`
- `generate_feature_tree`
- `generate_page_structure`
- `generate_wireframe`
- `generate_ui`
- `generate_code`
- `generate_tests`
- `deploy_project`

Skill 输出应统一为结构化结果，不直接只返回长文本。

## 8. Project Graph 设计建议

建议每个阶段都产出节点并建立边：

```text
Requirement -> PRD -> Feature -> Page -> Wireframe -> UI Spec -> Code Artifact -> Test Artifact -> Deploy Artifact
```

这样能实现：

- AI 仅读取相关节点
- 只更新变化节点
- 页面、功能、代码、测试可互相追踪

## 9. 分阶段实施计划

### Phase 1: 建基础骨架

目标：

- 建立 `ProjectConfig + ProjectGraph + ProjectMemory`
- 新建项目创建流程
- 替换演示数据入口

完成标准：

- 打开应用后先进入项目创建页
- 创建项目后写入本地配置
- 主界面依赖真实项目上下文渲染

### Phase 2: 打通产品链路

目标：

- 支持需求录入
- 支持 PRD / Feature Tree / Page Structure
- 节点统一写入 graph

完成标准：

- 可以从需求生成 feature tree
- 可以查看 graph 中节点关系

### Phase 3: 打通设计链路

目标：

- 重构 wireframe 数据模型
- Page 与 Wireframe 绑定
- Wireframe 持久化为 JSON

完成标准：

- 用户能创建页面并编辑 wireframe
- 切换页面后可以恢复对应设计

### Phase 4: 打通开发链路

目标：

- 文件树读取真实项目
- 终端执行真实命令
- AI 生成代码可回写文件

完成标准：

- 可从 feature/page/wireframe 触发代码生成
- 生成结果真正写入项目目录

### Phase 5: 打通测试与运维链路

目标：

- 生成测试
- 执行测试
- 生成部署配置

完成标准：

- 可以看到测试结果与部署产物

### Phase 6: 做体验优化

目标：

- 拖拽体验优化
- Agent 任务队列
- 差量上下文和缓存
- 可视化 graph 面板

## 10. 优先级清单

### P0

- 项目创建页
- 统一项目状态中心
- Project Graph 基础结构
- 真实文件树
- 真实终端
- Wireframe JSON 持久化

### P1

- Requirement -> PRD -> Feature Tree
- Page Structure
- Agent 编排器
- Skill 执行框架
- 代码生成回写

### P2

- 测试生成与执行
- Docker / CI / 部署配置生成
- Graph 可视化
- 项目记忆与增量上下文优化

## 11. 是否需要全量推倒

不建议全量推倒重写。

建议保留：

- Tauri 容器
- React 应用基础
- 现有画布部分渲染逻辑
- 现有部分 Tauri tool command

建议重构：

- `App.tsx` 的模块组织方式
- 所有以演示数据驱动的页面
- AI 调用层
- store 结构
- 工作区执行链路

## 12. 第一阶段落地任务

如果下一步直接开工，建议先做这一批：

1. 新建 `projectStore`
2. 新建 `ProjectCreatePage`
3. 定义 `ProjectConfig / ProjectGraph / ProjectMemory` 类型
4. 将 `App.tsx` 改为“项目创建态 / 工作区态”双入口
5. 去掉 `App.tsx` 中硬编码的 feature tree 初始化
6. 为项目配置增加本地持久化
7. 把产品、设计、开发、测试、运维页面改成依赖真实项目上下文

## 13. 我建议的下一步

最稳妥的推进方式：

1. 先做 Phase 1
2. 再做 Phase 2 和 Phase 3
3. 最后接入真实开发、测试、部署执行层

如果你同意，我下一步可以直接开始做第一阶段改造，先把：

- 项目创建页
- 项目配置类型
- 统一项目 store
- `App.tsx` 入口重构

这一套落地出来。
