当前设计 vs 你想要的设计

差异不在“有没有 loop”，而在“谁定义 loop 的真实语义”。

现在的真实结构

共享内核 + sidecar 自带语义
主执行链仍然存在 `runAgentTurn → runRuntimeToolLoop` 负责多轮模型调用、工具执行、结果回灌和最终回答。
sidecar 又定义了一套宿主行为 provider stream、工具 allowlist、shell 执行、approval 接入都在 sidecar 里再做了一轮拼装。
协议并不统一 Anthropic 走结构化 tool call，OpenAI-compatible 仍大量依赖文本 / XML 解析，稳定性自然不一致。
工具目录表面一致，执行语义其实漂了 前端有 `powershell`，sidecar executor 却没完整支持；Windows shell 还踩了 `/bin/zsh`。
to be simplified

你想要的单内核

单一内核 + sidecar 只是宿主层
只有一个 runtime kernel 决定流程 所有 provider、tool、approval、question、tool result 都进入同一状态机和同一事件协议。
sidecar 只提供宿主能力 会话、持久化、WebSocket、审批 UI、文件 / shell / 网络执行这些留在 sidecar，但不再定义第二套业务语义。
provider 全部翻译成 canonical events 模型可以直接回答，也可以多轮调工具，也可以 ask user，但都走统一事件层。
工具集、风险分级、结果格式只保留一份 从根上减少“看起来能用、实际上不同宿主行为不一样”的问题。

你和当前设计的出入点

这四个地方，就是你觉得“怎么越来越不对”的核心来源。

出入点 1

你记得的是“单一事实来源”

现在不是没有内核，而是共享内核外面又包了一层 sidecar 语义。用户感觉不到，但实现复杂度会立刻上升。

出入点 2

你希望 AI 自己决定路径

这点你是对的。合理设计不该把“思考 → 工具 → 反馈”写死成脚本，而该让模型在统一状态机里自然选择。

出入点 3

你要的是“常态上是 loop，结构上不是硬编码”

正常任务大概率确实会反复走工具往返，但实现层应该支持直接回答、只读 burst、ask user、审批后继续等分支。

出入点 4

你希望能顺手清理无用代码

如果决定回到单内核，这次就不该只修 bug，而要一起删掉 sidecar 中重复的 allowlist、重复协议和漂移的执行分支。

当前方案的优缺点

不是全坏，它解决了不少宿主层问题；但它把“运行时真相”拆成两份了。

优点

  • 共享内核仍然保留了真正的 agent loop，不是退化成一次性 request-response。
  • sidecar 带来了独立宿主进程，后续做持久化、审批、回放、后台任务都更顺手。
  • 现在已经具备 thinking、tool timeline、question、approval 这些比较完整的交互基建。
  • 只读工具的流式提前执行做得不差，体验上比纯串行“等模型说完再跑工具”更自然。

缺点

  • provider 协议分裂，Anthropic 和 OpenAI-compatible 的 tool 行为稳定性不一致。
  • 工具定义和执行宿主漂移，最典型的就是 Windows shell 和 `powershell` 支持不一致。
  • sidecar 同时承担宿主层和部分运行时语义，后面每改一次都容易再长分支。
  • 调试成本高，因为“问题在内核还是在 sidecar 拼装层”不再一眼能看出来。

改完以后会是什么状态

不是把 loop 砍掉，而是把 loop 的拥有权收回来。

Before

看起来是一个系统

但 provider、tool protocol、allowlist、executor、approval glue 分散在共享层和 sidecar 层。

After

真正单内核

所有模型事件先变成 canonical events，再由同一个 kernel 决定是否继续思考、调工具、问用户或结束回答。

Cleanup

删掉重复和漂移代码

sidecar 只保留 transport / persistence / execution host，不再维护第二套 runtime 业务语义。

改后的正常体验依旧会经常表现成“思考 → 执行 → 反馈 → 再执行 → 最终回答”，只是这将成为模型在统一状态机里的自然结果，而不是被多层代码勉强拼出来的既视感。
