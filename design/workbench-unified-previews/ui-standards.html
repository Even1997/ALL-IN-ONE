Foundation Standard

GoodNight Workbench UI Standard

Notes-first document space, Finder-like structure, and a native desktop tone. This page is the reusable baseline for tokens, icons, components, and interaction rules.

Quick Read

How To Use This Standard

1. Read the anatomy first

Understand the shell before touching cards, buttons, or accents.

2. Reuse the component cards

Rail, sidebar, document, and companion are the primary reusable building blocks.

3. Match the right board recipe

AI, graph, and overview pages each have a different center of gravity.

4. Feed the contract to AI

Use the machine-readable section at the end when asking AI to generate pages.

Workbench Anatomy

One Glance Layout Model

Topbar: location, search, global actions
Rail primary mode switch
Sidebar thread list, directory tree, source list, project list
Main Stage note surface, infinite canvas, or current working surface
Companion AI, inspector, quick actions, context
Status Bar: lightweight runtime or context facts
Rail

Always compact and icon-led. It changes the mode, not the content details.

Sidebar

List-oriented. Behaves like Notes or Finder source lists and directory trees, not like stacked cards.

Main Stage

The visual center. Usually a note surface or an infinite canvas, with only one dominant work mode at a time.

Companion

Secondary help, never a second competing main stage.

Core Rules

Direction Cards

Document First

The center of the product should feel like a calm note surface before it feels like a tool cluster.

  • One primary reading column in the main stage
  • AI suggestions live inline or in the right companion pane
  • Fewer bubbles, fewer pills, fewer nested cards
Desktop Native

Structure should come from rail, sidebars, toolbars, and status bars, not from dashboard decoration.

  • Pane-based layout stays consistent across pages
  • Subtle surface contrast instead of loud section framing
  • Rounded corners are restrained, not bubbly
AI Companion

AI should feel integrated with work, closer to Notes smart assistance than a standalone chat product.

  • Quick actions beat oversized response cards
  • Thinking reads like a margin note, not the hero section
  • Right pane summarizes, suggests, and helps navigate

AI Message And Card Standard

Lane Order Contract
User input Thinking Tool execution Final answer Confirm / next step
  • Render process states as separate lanes, never as one merged chat blob
  • Thinking is transient process context and must not become the final body
  • Tool execution uses runtime facts, not rewritten assistant prose
  • Final answer is the only durable reading surface for the completed turn
Dark Theme Preview

  • Do not invent a separate dark-mode structure for AI lanes
  • Dark mode lowers glare but preserves spacing, order, and card roles
  • Keep text contrast high and borders soft, not glowing
User Input
User prompt Compact source record

Keep the shell, add AI states, motion, and future-facing standards.

  • Compact and direct, closer to a note row than a message bubble
  • Should anchor the turn without overpowering the final answer
  • Use short metadata only when it helps trace context
Thinking Note
Thinking Transient process context

Mapping the page structure and placing AI states into reusable cards

  • Render like a quiet margin note, not like a hero reply
  • Keep it short, mutable, and visually secondary
  • Never persist thinking as the final answer body
Tool Execution
Tool run Runtime facts only

Read standard page and shared CSS

  • Use deterministic labels and stable runtime facts
  • Show progress, status, or result count without oversized loaders
  • Do not restyle tool output as assistant narrative
Final Answer
Final Durable answer body
  • This is the main reading surface for the finished turn
  • Allow more whitespace and cleaner typography than process cards
  • Keep it document-like, especially inside Notes-first layouts
Confirm Card
Confirm next step Decision before action

Apply this standard to AI chat and graph pages now?

  • Use for risky changes, branching choices, or destructive actions
  • Primary action is singular and obvious
  • Cards should feel actionable without looking like marketing CTAs
Blocked / Review
Needs review Polite escalation

Existing changes overlap this area. Review before continuing.

  • Use when human confirmation or conflict review is genuinely needed
  • Visual emphasis comes from tone and border, not loud danger fills
  • Always pair the issue with a clear next action
Motion System

Interactive States And Motion Rules

Hover / Focus 120-160ms, ease-out

Use a tiny lift, border tint, or background wash to signal hover and keyboard focus.

Press / Select 80-140ms, scale 0.985 or inset emphasis

Pressed states should feel precise and native, not bouncy or playful.

Loading / Reveal 180-240ms reveal, 900-1200ms pulse

Use motion only to explain state changes such as thinking, tool work, and confirmation.

Visual Heuristics

Good vs Avoid

Good Document-led hierarchy
  • one primary surface in the center
  • faint separators and restrained accent
  • icons replace repetitive labels where obvious
  • AI guidance sits inline or in the companion pane
Avoid Dashboard or chat-app drift
  • multiple equally loud cards competing at once
  • heavy pills, bubbles, and glowing action blocks
  • too many text buttons saying the same thing as icons
  • AI taking over the entire page as a feed

Execution Rules

Use / Avoid

Use
  • Pane hierarchy, toolbars, sidebars, and status bars
  • Icon plus short label instead of verbose controls
  • Whitespace to separate blocks before adding borders
  • One strong reading surface in the center of each work page
Avoid
  • Too many bubbles, pills, glow outlines, and floating cards
  • Repeated card-inside-card framing
  • Large brand color backgrounds or dashboard KPI clutter
  • Icon-only controls without meaning or hover help in real code

Prompt Snippet For AI 
Design a GoodNight desktop workbench page using a notes-first native macOS tone.
Keep the shell as: rail + sidebar + main stage + companion pane.
Make the center the dominant work surface, usually either a note surface or an infinite canvas.
Use directory-tree behavior for hierarchical file or source navigation.
Support both light and dark mode using the same hierarchy and component structure.
Render AI output as separate cards for user input, thinking, tool execution, final answer, and confirm-next-step when needed.
Include collapsed, expanded, empty, loading, selected, and error states for each major primitive.
Use subtle native-feeling motion for hover, selection, progress, and reveal.
Use restrained color, compact desktop density, and icon-led controls.
Avoid chat bubbles, KPI dashboards, pure black dark mode, and marketing-page composition.
Delivery Gate

Implementation Checklist

Page-Level Acceptance
  • Use the same shell order: rail, sidebar, main stage, companion pane
  • Main stage has one clear dominant surface and avoids multi-hero competition
  • Choose a primary workspace primitive intentionally: note surface, directory tree, or infinite canvas
  • Each primary primitive must define expanded, collapsed, empty, loading, selected, and error behavior where applicable
  • Icons replace repeated labels where the meaning is already obvious
  • Both light and dark themes share the same spacing, hierarchy, and card roles
  • No decorative gradients, floating bubbles, or dashboard KPI framing
AI Zone Acceptance
  • AI states are rendered as distinct cards, not a single merged chat blob
  • Thinking, tool execution, feedback, and final answer stay semantically separated
  • Every interactive card has default, hover, active, selected, collapsed, loading, and disabled rules
  • All motion has a meaning and is disabled or reduced cleanly under reduced-motion
  • Confirm cards appear only for real decisions, not for every assistant suggestion
