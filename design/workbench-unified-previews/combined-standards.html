How To Use This Standard

Understand the shell before touching cards, buttons, or accents.

Workbench Anatomy

One Glance Layout Model

Topbar: location, search, global actions
Rail primary mode switch
Sidebar thread list, directory tree, source list, project list
Main Stage note surface, infinite canvas, or current working surface
Companion AI, inspector, quick actions, context
Status Bar: lightweight runtime or context facts
Rail

Always compact and icon-led. It changes the mode, not the content details.

Sidebar

List-oriented. Behaves like Notes or Finder source lists and directory trees, not like stacked cards.

Main Stage

The visual center. Usually a note surface or an infinite canvas, with only one dominant work mode at a time.

Companion

Secondary help, never a second competing main stage.

Core Rules

Direction Cards

Document First

The center of the product should feel like a calm note surface before it feels like a tool cluster.

  • One primary reading column in the main stage
  • AI suggestions live inline or in the right companion pane
  • Fewer bubbles, fewer pills, fewer nested cards
Desktop Native

Structure should come from rail, sidebars, toolbars, and status bars, not from dashboard decoration.

  • Pane-based layout stays consistent across pages
  • Subtle surface contrast instead of loud section framing
  • Rounded corners are restrained, not bubbly
AI Companion

AI should feel integrated with work, closer to Notes smart assistance than a standalone chat product.

  • Quick actions beat oversized response cards
  • Thinking reads like a margin note, not the hero section
  • Right pane summarizes, suggests, and helps navigate

Confirm next step Decision before action

Apply this standard to AI chat and graph pages now?

  • Primary action is singular and obvious
  • Cards should feel actionable without looking like marketing CTAs
Blocked / Review
Needs review Polite escalation

Existing changes overlap this area. Review before continuing.

  • Use when human confirmation or conflict review is genuinely needed
  • Visual emphasis comes from tone and border, not loud danger fills
  • Always pair the issue with a clear next action
Motion System

Interactive States And Motion Rules

Hover / Focus 120-160ms, ease-out

Use a tiny lift, border tint, or background wash to signal hover and keyboard focus.

Press / Select 80-140ms, scale 0.985 or inset emphasis

Pressed states should feel precise and native, not bouncy or playful.

Loading / Reveal 180-240ms reveal, 900-1200ms pulse

Use motion only to explain state changes such as thinking, tool work, and confirmation.

Visual Heuristics

Good vs Avoid

Good Document-led hierarchy
  • one primary surface in the center
  • faint separators and restrained accent
  • icons replace repetitive labels where obvious
  • AI guidance sits inline or in the companion pane
Avoid Dashboard or chat-app drift
  • multiple equally loud cards competing at once
  • heavy pills, bubbles, and glowing action blocks
  • too many text buttons saying the same thing as icons
  • AI taking over the entire page as a feed

Delivery Gate

Implementation Checklist

Page-Level Acceptance
  • Use the same shell order: rail, sidebar, main stage, companion pane
  • Main stage has one clear dominant surface and avoids multi-hero competition
  • Choose a primary workspace primitive intentionally: note surface, directory tree, or infinite canvas
  • Each primary primitive must define expanded, collapsed, empty, loading, selected, and error behavior where applicable
  • Icons replace repeated labels where the meaning is already obvious
  • Both light and dark themes share the same spacing, hierarchy, and card roles
  • No decorative gradients, floating bubbles, or dashboard KPI framing
AI Zone Acceptance
  • AI states are rendered as distinct cards, not a single merged chat blob
  • Thinking, tool execution, feedback, and final answer stay semantically separated
  • Every interactive card has default, hover, active, selected, collapsed, loading, and disabled rules
  • All motion has a meaning and is disabled or reduced cleanly under reduced-motion
  • Confirm cards appear only for real decisions, not for every assistant suggestion

State Contract

One Shared Language

Required States
default hover selected collapsed empty loading error
  • These states must be designed intentionally, not patched in later
  • Every major workbench surface should use the same visual logic for status and emphasis
  • Selection is stronger than hover, but should still feel quiet and desktop-native
Behavior Rules
defaultCalm resting state with clear hierarchy and no unnecessary emphasis.
hoverSubtle tint, shadow, or lift only. Never use loud glow or bubble chrome.
selectedPersistent focus state for the current file, node, note, or AI result.
collapsedKeep identity, summary, and status. Remove depth, not meaning.
emptyQuiet starter state with one next action and clear context.
loadingUse pulse, skeleton, or progress. Avoid decorative shimmer walls.
errorExplain the issue inline and provide the recovery path inside the same surface.
Motion Rules
Default rest
Hover 120-160ms
Selected persistent
Error and empty stay static
  • Hover and select motion should feel Finder / Notes-like, not playful
  • Loading uses pulse or progress only, while error and empty rely on clarity rather than motion
  • Dark mode must preserve the same interaction hierarchy, not invent a different one
