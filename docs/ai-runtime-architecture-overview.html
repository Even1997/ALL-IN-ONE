AI Runtime / 总览优先

当前 AI 架构总览

这套 AI 主链路可以压缩成一句话：AIChat 发起用户回合 → Runtime 编排与工具循环 → Provider/Sidecar 输出运行时事实 → Canonical/Timeline 映射 → Render Model 组装 → UI 分 lane 展示。 这页重点不是“理想架构”，而是把仓库里现在真正跑着的层、模块、链路和排查入口收拢到一张页面里。

8 层 从 UI 入口到 Rust / Node sidecar / 协议与持久化
4 条实际运行路线 Built-in、Provider 壳层 Claude/Codex、Local Claude CLI、Local Codex CLI
4 个核心边界 Provider 适配 → Canonical 事件 → Timeline / Projection → Render / UI

一、分层总览

先看“谁负责什么”，再往下看“数据怎么流”。这里按当前仓库里的真实边界来分，而不是按抽象概念重命名。

L1 / UI 入口层

AIChat 与工作台装配

负责把会话、输入、消息列表、运行时交互、设置与 sidecar 动作拼成完整聊天工作台。
  • 关键文件：src/components/workspace/AIChat.tsx
  • 作用：页面级总入口，连聊天 store、runtime store、approval、replay、sidecar
  • 排查入口：消息为什么没发出去、会话为什么没切对、页面为什么没联动
L2 / 交互编排层

Chat Turn 协调与回合总入口

负责一次用户提交后的主链路编排：建 session、路由 provider、工具、审批、流式草稿与收口。
  • 关键文件：src/modules/ai/runtime/orchestration/runtimeChatTurnCoordinator.ts
  • 作用：前端侧 runtime turn 的总协调入口
  • 排查入口：某一轮 turn 为什么走 built-in / mcp / local agent，为什么被审批或中断
L3 / Agent Kernel 层

Built-in Agent 执行与工具循环

负责真正执行 built-in agent：上下文构建、工具循环、技能准备、兜底重试与最终文本收口。
  • 关键文件：src/modules/ai/runtime/orchestration/executeRuntimeBuiltInAgentTurn.ts、src/modules/ai/runtime/agent-kernel/runAgentTurn.ts、src/modules/ai/runtime/tools/runtimeToolLoop.ts
  • 作用：把“模型 + 工具 + 技能 + 上下文”串成一轮完整执行
  • 排查入口：为什么没调工具、为什么 tool loop 耗尽、为什么 final 被兜底改写
L4 / Provider 适配层

Provider 协议适配与原生流解析

负责请求 OpenAI / Anthropic / 兼容 OpenAI，并把各家的原生流统一成 runtime provider 事件。
  • 关键文件：apps/runtime/src/nodeRuntimeProviderClient.ts、src/modules/ai/runtime/provider/runtimeProviderEvents.ts
  • 作用：只做协议选择、SSE/JSON 解析、tool call 装配、usage 与 reasoning 透传
  • 排查入口：thinking 为什么没出来、Responses / Chat Completions 怎么切、tool args 为什么拼错
L5 / Canonical 真相层

Runtime 事实标准化

负责把 provider 原生事件、assistant timeline、sidecar snapshot 映射成统一 canonical 事件与 timeline 事实。
  • 关键文件：src/modules/ai/store/assistantTimeline.ts、src/modules/runtime-sidecar/runtimeSidecarCanonical.ts、packages/runtime-protocol/src/index.ts
  • 作用：把 thinking / text / tool / approval / question 分开建模
  • 排查入口：正文为什么和 tool lane 混了、为什么后端有事件前端没显示
L6 / Sidecar 桥接层

Desktop / Node Runtime Sidecar 桥

负责前端与 node runtime sidecar 之间的连接、事件订阅、snapshot 映射和动作回传。
  • 关键文件：src/modules/runtime-sidecar/desktopRuntimeSidecar.ts、src/modules/runtime-sidecar/runtimeSidecarSessionBridge.ts、apps/runtime/src/index.ts
  • 作用：把 runtime sidecar 的会话真相接入前端 store
  • 排查入口：前端为什么连不上 sidecar、snapshot 为什么没落到 chat store
L7 / Projection 与 Render 层

Render Model 与输出模型

负责把 assistant timeline 压成 thinking / feedback / answer 三种可展示 lane，再组装成 UI 输出。
  • 关键文件：src/components/workspace/assistantRenderModel.ts、src/components/workspace/assistantMessageOutputModel.ts
  • 作用：保证 thinking 是过程、final answer 是最终正文、tool 卡片来自 runtime truth
  • 排查入口：为什么正文被拼接、为什么 thinking 和 final 顺序不对、为什么 approval-only 消息不显示
L8 / 状态与持久化层

Chat Store / Runtime Store / Rust 命令层

负责聊天视角、执行视角、sidecar 连接态以及桌面端持久化命令。
  • 关键文件：src/modules/ai/store/aiChatStore.ts、src/modules/ai/runtime/agentRuntimeStore.ts、src/modules/ai/runtime/agentRuntimeClient.ts、src-tauri/src/agent_runtime/commands.rs
  • 作用：分别保存聊天展示真相、运行现场真相、桌面侧数据读写
  • 排查入口：刷新后为什么丢、timeline 为什么没持久化、审批 / memory / checkpoint 为什么没写入
当前架构最重要的约束是：不要把 UI 展示策略下沉到 provider adapter 或 canonical 真相层。 thinking、tool、feedback、final 必须继续分开，最终正文只应该来自 final answer。

二、关键模块

下面这些文件是现在最常需要回看的骨架节点。它们不是“所有模块”，而是对理解主链路最关键的一组。

1. AIService（旧总入口 / 仍在 Provider 路线上扮演关键角色）

  • 文件：src/modules/ai/core/AIService.ts
  • 作用：旧的通用 AI 调用入口，仍承接 provider 请求、流式回调、工具协议识别与部分执行辅助
  • 现在的位置：更像“服务编排层”，不是单纯 provider adapter

2. Built-in Runtime 执行器

  • 文件：src/modules/ai/runtime/orchestration/executeRuntimeBuiltInAgentTurn.ts
  • 作用：built-in agent 的完整回合执行器，是 runtime turn 的主执行器
  • 特点：这里会处理技能、memory、工具循环、失败兜底与最终内容标准化

3. Tool Loop

  • 文件：src/modules/ai/runtime/tools/runtimeToolLoop.ts
  • 作用：解析 assistant 输出中的工具调用、执行工具、压缩上下文、继续回合
  • 典型问题：tool protocol 解析失败、重复 tool call、context compaction、loop exhausted

4. Node Runtime Provider Client

  • 文件：apps/runtime/src/nodeRuntimeProviderClient.ts
  • 作用：当前 built-in GPT / OpenAI-compatible / Anthropic 的真正 provider 流入口
  • 近期重点：官方 OpenAI 优先走 /responses，兼容 provider 继续走 /chat/completions

5. Sidecar Session Bridge

  • 文件：src/modules/runtime-sidecar/runtimeSidecarSessionBridge.ts
  • 作用：把 sidecar snapshot / event 同步到 chat store、runtime store、approval store
  • 典型问题：后端明明有事件，但前端消息没动；approval / question 没闭环

6. Assistant Timeline / Render Model

  • 文件：src/modules/ai/store/assistantTimeline.ts、src/components/workspace/assistantRenderModel.ts
  • 作用：前者维护消息里的运行时事实，后者决定最终 UI 如何分 lane 展示
  • 典型问题：正文被拼接、thinking 跑到 final 里、tool / approval 卡片顺序错

7. Claude / Codex Runtime 壳层

  • 文件：src/modules/ai/gn-agent/runtime/claude/ClaudeRuntime.ts、src/modules/ai/gn-agent/runtime/codex/CodexRuntime.ts
  • 作用：把 GN agent 运行时映射到当前配置，并复用 AIService 执行 prompt
  • 意义：Claude / Codex 是 provider 壳层，不是另一套完全独立的 UI 体系

8. Tauri Commands

  • 文件：src-tauri/src/agent_runtime/commands.rs
  • 作用：桌面端持久化命令入口，处理 thread、timeline、approval、memory、checkpoint、settings 等
  • 典型问题：桌面侧数据没写进去、列表读不到、跨会话信息消失

四、把所有链路放到一个例子里

假设用户问一句：“帮我分析这个项目的 AI 聊天为什么正文会被拼接。”同一个问题，根据入口模式不同，会走下面 4 条路线之一。

    B -->|"classic / built-in"| C1["runtimeChatTurnCoordinator"]
    C1 --> D1["executeRuntimeBuiltInAgentTurn"]
    D1 --> E1["runAgentTurn / runtimeToolLoop"]

    F1 --> G1["Anthropic 或 OpenAI-compatible 协议流"]

    C2 --> D2["ClaudeRuntime.ts"]
    D2 --> E2["AIService.complete*"]
    E2 --> F2["Anthropic app-config 调用"]
    F2 --> H

    B -->|"providerExecutionMode = codex"| C3["agentRuntimeClient.executePrompt"]
    C3 --> D3["CodexRuntime.ts"]
    D3 --> E3["AIService.complete*"]
    E3 --> F3["OpenAI-compatible app-config 调用"]
    F3 --> H

    B -->|"local agent flow"| C4["runtimeLocalAgentFlow"]
    C4 --> D4["Tauri 命令层"]

五、Built-in / Claude / Codex 到底怎么分

这是这次最需要澄清的一点，直接按当前代码结论写死：

1. Built-in 不是只支持 OpenAI-compatible

  • 配置模型来源：src/modules/ai/store/aiConfigState.ts
  • 当前 provider 类型只有两个：openai-compatible、anthropic
  • Built-in 的 provider 适配在：apps/runtime/src/nodeRuntimeProviderClient.ts
  • 结论：built-in 这条线本身同时支持 Anthropic 和 OpenAI-compatible

2. ClaudeRuntime 是“Anthropic 壳层”，不是本地 Claude CLI

  • 文件：src/modules/ai/gn-agent/runtime/claude/ClaudeRuntime.ts
  • 它要求配置必须是 provider === 'anthropic'
  • 它内部仍然调用：AIService.completeText / completeMessages
  • 结论：它现在更像“Claude 模式下的 app-config 执行壳层”

3. CodexRuntime 是“OpenAI-compatible 壳层”，不是本地 Codex CLI

  • 文件：src/modules/ai/gn-agent/runtime/codex/CodexRuntime.ts
  • 它要求配置必须是 provider === 'openai-compatible'
  • 它内部也复用：AIService.completeText / completeMessages
  • 结论：它是 Codex 模式下的 app-config 执行壳层，不等于本地 codex 终端

4. 本地 Claude / Codex CLI 是另一条 local agent 链

  • 前端流控：src/modules/ai/runtime/orchestration/runtimeLocalAgentFlow.ts
  • Tauri 命令：src-tauri/src/lib.rs
  • 真实命令：claude -p、codex exec ... --output-last-message
  • 结论：本地终端版 Claude/Codex 是独立于 ClaudeRuntime.ts / CodexRuntime.ts 的另一条链
