1. 整体架构概览

GoodNight 的 AI 系统采用分层架构，分为表现层、编排层、运行时和后端持久化四层。各层通过 Zustand store 和 Tauri invoke 通信。

🎨 表现层 (Presentation)
React 组件：Workspace → Chat → MessageViewport / ComposerToolbar，以及各类面板（Memory、Timeline、Skills、Context、Approval）
🔀 编排层 (Orchestration)
负责路由用户意图：Direct Chat Flow → Local Agent Flow → Built-in Agent Turn → Team Flow，协调审批、上下文组装、Memory 提取
⚙️ 运行时层 (Runtime)
Zustand Store（agentRuntimeStore）+ 客户端 API（agentRuntimeClient）+ 工具循环（runtimeToolLoop）+ Provider 执行
🦀 持久化层 (Persistence)
Rust Tauri 后端：agent_runtime/（线程、Timeline、Memory、审批等 JSON 文件存储）+ agent_shell/（Shell 会话、设置）

核心设计原则

  • 事件驱动：所有状态变更通过 Zustand store 的 actions 触发，组件通过 selector 订阅
  • Provider 抽象：Built-in、Claude、Codex 三种 Provider 通过统一接口执行，由 Runtime Registry 解析
  • 可恢复性：Turn 执行记录 Replay Event，支持断点恢复、重试和回放
  • 审批门禁：高风险操作（bash、remove）需要用户审批，通过 runtimeApprovalCoordinator 协调
关键路径提示：用户的一条消息最终会经过 "组件 → Store → 编排 → 上下文组装 → Provider 调用 → 工具循环 → Memory 提取 → Timeline 记录" 的完整链路。

4. 编排层（Orchestration）

编排层位于 src/modules/ai/runtime/orchestration/，负责将用户意图路由到正确的执行路径并管理生命周期。

编排核心文件

runtimeDirectChatFlow.ts
直接对话流的请求构建与响应归一化。使用 buildRuntimeDirectChatRequest() 组装 Prompt，normalizeRuntimeDirectChatResponse() 清理内部标记。
executeRuntimeBuiltInAgentTurn.ts
内置代理的完整 Turn 执行。包含 Skills 准备、Direct Chat 构建、runAgentTurn() 调用、响应归一化、Project Access 失败降级、Artifact Draft 重试、文件变更守卫、Memory 提取。
runtimeLocalAgentFlow.ts
本地代理流程。风险分类（Risk Policy）→ 审批请求 → 执行 CLI 命令 → 决策处理回调。
runtimeTurnSessionFlow.ts
Session 状态机辅助函数：applyRuntimeTurnClassifying()、buildRuntimeTurnReviewPlan()、applyRuntimeTurnExecuting/Completed/Failed/Blocked()。
runtimeTurnOutcomeFlow.ts
创建 Activity Log 记录和标准化的 Outcome 描述（成功/失败），用于 Built-in 和 Local Agent。
agentTurnRunner.ts
Turn 和 Execution 控制器的工厂函数，提供 Streaming 消息组装器。包含 Turn ID 生成、createRuntimeTurnController()、createRuntimeExecutionController()、流式消息状态机。

Conversation Gateway（对话网关）

runtimeConversationGateway.ts + useRuntimeConversationGateway.ts 是一个重要的汇聚点，它将 aiChatStore 和 agentRuntimeStore 的数据合并为一个 RuntimeConversationProjection，组件通过 React Hook 订阅。包含：最新的 Turn Session、Live State、审批、Memory Candidates、Replay 状态、Team Runs 等。

5. Turn 执行生命周期

Session 状态机

一个 Turn 从创建到完成经历以下状态转换：

queued → classifying → planning → waiting_approval → executing → completed → failed → resumable

状态转换定义在 agentSessionStateMachine.ts 的 reduceAgentTurnSession() 纯函数中，事件类型有 7 种：start_classifying → enter_planning → plan_waiting_approval → approval_granted → execution_blocked → execution_completed → execution_failed。

Agent Kernel 执行

1
构建上下文
buildAgentContext(input) — 将项目信息、Instructions、Skills、Memory、Reference Files、对话历史组装为带预算的 AgentContextSnapshot
2
运行工具循环
runRuntimeToolLoop() — 调用模型 → 解析 Tool Call → 执行工具 → 结果回传 → 重复（最多 4 轮）。支持 Streaming 检测工具调用、并行执行只读工具、上下文超长时自动压缩。
3
返回结果
返回 RunAgentTurnResult：finalContent（最终文本）+ context（上下文快照）+ toolCalls（工具调用记录）+ transcript（完整消息日志）

工具循环（Tool Loop）

runtimeToolLoop.ts 是整个 AI 代理的核心推理循环：

  • 流式检测：createStreamingToolDetector() 在流式输出中实时检测 <tool_use> 标记
  • 工具解析：parseToolCalls() 解析 XML 格式的工具调用
  • 并行执行：只读工具（glob、grep、ls）并行执行，写操作串行
  • 压缩机制：上下文超长时触发 compactOldToolResults() 或 removeOldestTurn()
  • 结果预览：工具结果截断至 1000 字符，防止 Token 溢出
  • 文件变更追踪：提取工具执行的 File Changes，供审批守卫校验
系统提示词：Agent Kernel 的系统提示词直接硬编码在 runAgentTurn.ts 中（GOODNIGHT_AGENT_SYSTEM_PROMPT），规定了可用工具、XML 格式、内部文件保密规则、回答策略等。允许的工具包括：glob, grep, ls, view, write, edit, bash, fetch, AskUserQuestion。

Streaming 消息组装

createRuntimeStreamingMessageAssembler() 是一个状态机，将 AI 的流式输出片段逐步组装为结构化的 Draft，包含 thinking 和 text 两部分，并做可见文本消毒。

6. 上下文组装系统

上下文组装位于 src/modules/ai/runtime/context/，负责将项目信息、历史记录和参考材料组装为发给 AI 的完整 Prompt。

assembleAgentContext.ts
简单工厂函数，从 Project 信息、Instructions、Reference Files、Memory、Active Skills 创建 AgentContextBundle。
buildThreadPrompt.ts
将 AgentContextBundle 渲染为 XML 格式的 Prompt 字符串，含 <instructions>、<skills>、<memory>、<references> 等标签。
buildAgentContext.ts
核心构建器。将输入分解为 7 个 Section：instructions, skills, history, memory, reference, active-context, user-input。通过 allocateContextBudget() 做贪心分配，最多迭代 8 次收敛 Token 计数。
contextBudgetAllocator.ts
贪心预算分配：Required（instructions + user-input）始终包含，Optional 按优先级依次包含直到 Token 上限。
agentContextTypes.ts
类型定义：AgentContextSection、AgentContextBudget、AgentContextBuildInput、AgentContextSnapshot、AgentContextSectionKind。

上下文结构

构建后的 Prompt 是一个多层 XML 文档，末尾附有 <context_report> 报告 Token 使用情况：

1
Instructions（必须）
系统指令和行为规则，如 FILE_OPERATION_TRUTHFULNESS_POLICY、TASK_AUTHORIZATION_POLICY 等
2
Skills
3
History | Memory | Reference | Active Context
按优先级排队，预算内尽量包含。Active Context 包含 project + project_id + thread_id
4
User Input（必须）
当前用户的输入消息
5
<context_report>
used_tokens / limit_tokens 报告，列出 included 和 excluded 的各 Section 及 Token 数

7. 审批与风控系统

审批系统位于 src/modules/ai/runtime/approval/，为高风险操作提供门禁。

riskPolicy.ts
风险分类：
🔴 High：tool_remove、tool_bash、run_local_agent_prompt
🟡 Medium：tool_edit、tool_write、project_file_write
还根据敏感路径（.env、package.json、src-tauri/）提升风险等级。
approvalStore.ts Store
Zustand Store：approvalsByThread、sandboxPolicy、permissionMode。Actions：enqueueApproval()、resolveApproval()。
runtimeApprovalCoordinator.ts
核心协调器：requestRuntimeApproval() 创建审批记录并在 pendingApprovalActions 存储回调；resolveRuntimeApproval() 查找并执行回调。
permissionMode.ts
策略映射：'ask' → 每次询问、'plan' → 先计划再执行、'auto' → 自动执行、'bypass' → 跳过审批。
approvalTypes.ts
类型：ApprovalRiskLevel ('low'|'medium'|'high')、ApprovalStatus、SandboxPolicy、PermissionMode、ApprovalRecord。

8. Provider 运行时

系统支持三个 AI Provider，通过统一的注册机制集成。

Provider 注册

🏠 Built-in（内置）
AIService.ts 单例，支持 Anthropic 和 OpenAI-compatible 两种 API。包含完整的 runAgentLoop()（多轮工具调用）、Streaming SSE、内置 Tool Executor。
src/modules/ai/core/AIService.ts
🟣 Claude
ClaudeRuntime 类，包装 aiService.completeText()，筛选 provider === 'anthropic' 的配置。检测本地 .claude/settings.json。
ClaudeRuntime.ts
🔵 Codex
CodexRuntime 类，和 ClaudeRuntime 结构一致，筛选 provider === 'openai-compatible'。检测本地 .codex/ 目录。
CodexRuntime.ts

Provider 配置

  • Provider Presets（providerPresets.ts）：预置 7 个 Provider：MiniMax、SiliconFlow、OpenRouter、DeepSeek、Anthropic、OpenAI、Ollama
  • AI Config State（aiConfigState.ts）：管理 AIConfigEntry 的增删改查、Preset 合并、启用过滤
  • Runtime Registry（runtimeRegistry.ts）：运行时解析 Chat Agent：'claude' | 'codex' | 'team' | 'built-in'

9. 技能（Skills）系统

Skills 系统是 AI 的能力扩展机制，支持从本地文件和系统捆绑两个来源加载，按需激活。

Skills 生命周期

1
发现与加载
skillLibrary.ts 通过 discover_local_skills Tauri 命令发现本地 SKILL.md 文件，用 parseSkillMarkdown.ts 解析 frontmatter + body，与系统捆绑 Skill 合并去重。
2
意图路由
3
注册与管理
runtimeSkillRegistry.ts 管理每个 Thread 的 Skill 激活列表，支持 activationPaths 自动激活（基于 glob 匹配）、无条件激活和 Thread 级手动激活。
4
Prompt 生成
buildRuntimeSkillPrompt.ts 将激活的 Skill 定义渲染为 XML <skill> 块，包含元数据属性。
5
预处理
runtimeSkillPreparation.ts 在 Turn 执行前预处理 Skill：参数替换（substituteRuntimeSkillArguments）、Shell 变量替换（${CLAUDE_SESSION_ID}）、执行内联 Shell 命令（!cmd`` 或 ```! 代码块）、创建 Tool Hook Runner（PreToolUse/PostToolUse）。

Skill 定义结构

RuntimeSkillDefinition 包含：id、name、description、whenToUse、prompt、token、aliases、executionContext（'inline'|'fork'）、agent、model、hooks、activationPaths、allowedTools、userInvocable、modelInvocable、source。

13. 完整请求链路

以下是用户发送一条消息后的完整执行链路，涵盖所有系统和子步骤。

1
用户输入消息
ComposerToolbar 提交文本 → aiChatStore 追加消息到 ChatSession
→ resolveSkillIntent() 检查是否触发 /skill 或 @skill
2
创建 Turn & Session
createQueuedRuntimeTurn() → startRuntimeTurnLifecycle() → submitTurn()
→ agentSessionStateMachine: classify → plan → execute
3
路由到执行路径
根据 Provider 类型和消息上下文选择：
Built-in Turn → executeRuntimeBuiltInAgentTurn.ts
Direct Chat → buildRuntimeDirectChatRequest()
Local Agent → prepareRuntimeLocalAgentFlow() → 风险分类 → 审批
4
上下文组装
buildAgentContext() → 7 个 Section 构建 → allocateContextBudget() 贪心分配 → 最多 8 次迭代收敛 Token → 生成 <context_report>
5
Agent Kernel 执行
runAgentTurn() → runRuntimeToolLoop():
→ 调用 AI 模型（Streaming）
→ 流式检测工具调用标记
→ 解析 XML Tool Call
→ 并行执行只读工具 / 串行执行写工具
→ 结果回传到模型
→ 重复直到模型返回最终内容或达到 4 轮上限
6
响应后处理
normalizeRuntimeDirectChatResponse() 清理内部标记
重试机制：Project Access 失败降级 / Artifact Draft 重试
runtimeFileMutationClaimGuard.ts 校验未验证的文件变更声明
extractMemoryCandidates() 提取记忆候选
7
持久化与更新
Turn 状态 → completed / failed / resumable
Timeline 事件追加
Tauri invoke → Rust Store → JSON 文件
Conversation Gateway 触发组件更新 → 渲染新消息

重试与恢复机制

🔄 Project Access Fallback
如果内置代理的 Project Context 访问失败，自动降级重试 — 不使用工具，仅基于用户输入回答。
📝 Artifact Draft Retry
如果模型仅返回流程叙述而非实际内容，自动重试一次。
