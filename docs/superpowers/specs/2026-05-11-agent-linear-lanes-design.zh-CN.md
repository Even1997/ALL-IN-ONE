# Agent Linear Lanes 设计

## 摘要

这次调整不再只优化折叠态时间线，而是把 Agent 聊天里的三类主要内容统一成同一种更线性的阅读语言：

- 运行时间线
- 思考过程
- 正文回答

目标是让它们在视觉上都更接近参考图里的“过程日志流”，而不是几种不同的卡片混排。与此同时，需要兼顾右侧侧边栏较窄时的可读性，避免每一块内容都依赖宽卡片才能成立。

## 目标

- 让时间线、思考、正文都从“面板/标签卡片”转向“线性 lane”表达。
- 在较窄侧边栏中仍然可读，不因为 padding、圆角、并列元信息过多而显得拥挤。
- 保留现有交互能力：
  - 思考块仍可展开/收起
  - 时间线详情仍可展开/收起
  - 正文仍支持 Markdown、流式输出和内联图片
- 统一三者的节奏：
  - 左侧引导
  - 单列主阅读线
  - 弱化的辅助元信息

## 非目标

- 不改 runtime timeline 的数据结构、排序或聚合逻辑。
- 不改 assistant timeline 的事件生成逻辑。
- 不重做用户消息气泡样式。
- 不修改详情抽屉、审批卡片、提问卡片的数据内容。

## 参考图拆解

参考图更像“连续过程记录”而不是“小卡片列表”，关键特征有四个：

- 信息沿一条纵向阅读轴展开，而不是一块块独立容器。
- 左侧有轻量引导线，用来组织层级，而不是靠粗边框和大圆角。
- 主体内容是单列文本，状态和操作退居辅助位置。
- 即使有折叠，折叠后的头部也更像一句日志，而不是一个独立按钮卡片。

## 当前问题

当前实现虽然已经把时间线压缩成单行，但整体仍不够统一：

- `TimelineCard` 还是保留了较明显的“卡片剩余感”。
- `chat-thinking-block` 依然是一个完整的玻璃面板，和时间线的线性语言不一致。
- `chat-answer-text` 仍是厚 padding、大圆角、大面板的正文卡。
- assistant 内容区的 lane 宽度和间距仍偏宽，放在窄侧边栏里会显得松散且占空间。

## 设计决策

### 1. 把 assistant 输出视为统一的 lane 系统

运行时间线、思考块、正文块都属于 assistant 的同一条输出流，只是语义不同。因此视觉上应共享一套基础语言：

- 更窄的横向占用
- 更轻的背景存在感
- 更弱的容器边界
- 更强的纵向连续性

### 2. 时间线保持“单行日志”方向，但进一步弱化卡片感

时间线折叠态继续采用一行阅读模型，但要进一步向参考图靠拢：

- 左侧保留细竖线
- 主体内容保持单行截断
- 状态和“详情”保持弱文本样式
- 背景洗色更轻，避免再次形成条状卡片感

### 3. 思考块改成“可展开的日志行 + 内容体”

思考块应从“独立玻璃卡片”改成“可展开 lane”：

- 折叠时：
  - 只保留一行头部
  - 左侧轻量引导
  - `思考中/思考过程 + 时长 + 预览` 读成一句线性信息
- 展开时：
  - 内容体挂在该 lane 下方
  - 内容区比当前更克制，像附属明细而不是第二张大卡片

### 4. 正文块改成“正文 lane”，而不是大气泡卡

正文仍需要承载 Markdown，但其容器不应继续强调“独立面板”：

- 降低 padding、圆角、阴影和边框存在感
- 在常规段落状态下更接近连续正文
- 对代码块、表格、引用等复杂块保持必要区分
- 流式输出状态仍保留，但避免因为 streaming 态重新变成重卡片

### 5. 为窄侧边栏做自适应约束

本次样式要优先考虑窄宽度下的稳定性：

- 收紧 assistant lane 的最大宽度和内部 gap
- 降低左右 padding，减少横向浪费
- 元信息只保留一层，不允许多层并排挤压正文
- 长文案统一单行截断或自然换行，不让辅助元素抢宽度

## 测试策略

- 用 source-level 测试锁定 `TimelineCard` 仍是单行日志结构。
- 更新 CSS 回归测试，把思考块和正文块从“厚卡片 contract”改成“线性 lane contract”。
- 跑现有 timeline / runtime 输出相关测试，确认交互与内容流没有回归。

## 完成标准

- 时间线、思考、正文三类 assistant 输出在视觉上属于同一套语言。
- 窄侧边栏下没有明显的卡片堆叠感和横向拥挤感。
- 思考与时间线仍可折叠/展开，正文渲染能力不受影响。
- 相关 source-level 与 runtime 输出测试通过。
