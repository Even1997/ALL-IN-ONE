# AI Chat Markdown 阅读与保存到项目设计

## 摘要

这个设计为现有 AI 聊天补上两项能力：

- assistant 普通回答按 Markdown 渲染阅读
- assistant 普通回答可一键保存到当前项目，保存结果是新的知识库笔记，并同步 `.md` 镜像

这次改动坚持最小接入原则：复用现有 `KnowledgeMarkdownViewer`、知识库笔记创建链路和 Markdown 镜像写入逻辑，不新增复杂编辑流程，不改变现有 `thinking` / `tool` / `knowledgeProposal` 交互。

## 目标

- 让 AI assistant 的正文回答具备正常的 Markdown 阅读体验。
- 让用户可以把一条聊天回答直接沉淀为项目里的正式 Markdown 文档。
- 保存行为默认安全，不覆盖已有知识笔记，每次点击都新建一篇。
- 复用现有知识库和本地镜像能力，避免引入第二套文档保存机制。

## 非目标

- 不为用户消息或 system 消息增加保存入口。
- 不把聊天区改造成 Markdown 编辑器。
- 不增加“保存前手动改标题”“另存为路径选择”“覆盖当前笔记”等复杂交互。
- 不自动添加 `ai-chat`、`AI 生成` 等来源标签。
- 不改变现有知识提案卡片的审批和执行逻辑。

## 当前问题

当前 AI 聊天里的普通文本回答仍以纯文本分行方式显示：

- 列表、标题、引用、表格、代码块没有统一的 Markdown 阅读体验
- 聊天回答虽然常常本质上已经是可复用文档，但不能直接沉淀到项目知识库
- 现有“知识提案”链路只覆盖特定工作流产物，不覆盖普通 assistant 回答

结果是用户需要手动复制内容到别处，再自己整理成项目文档。

## 设计原则

- 只补用户明确要的能力，不顺手扩展更多编辑和管理功能。
- Markdown 阅读直接复用现有 viewer，不重造渲染器。
- 保存动作默认新建，优先避免误覆盖。
- 聊天里的思考块和工具块继续保持结构化呈现。
- 保存后的文档应被当作普通项目 Markdown 文档，而不是带有特殊身份的 AI 文档。

## 用户体验

### Markdown 阅读

assistant 消息继续先经过现有 `parseAIChatMessageParts` 解析：

- `thinking` 块保持现状
- `tool` 块保持现状
- `text` 块改为用 `KnowledgeMarkdownViewer` 渲染

因此一条回答如果同时包含思考、工具输出和正文，会继续按块展示，但正文部分拥有正常的 Markdown 阅读能力。

### 保存入口

仅在满足以下条件时，为 assistant 消息显示 `保存到项目` 按钮：

- 消息角色是 `assistant`
- 消息存在可保存的正文内容
- 当前不是流式输出中的临时草稿状态
- 当前项目已打开

用户点击后：

- 按钮进入 `保存中...`
- 成功后显示 `已保存到项目`
- 失败后显示错误信息，但不影响消息内容本身

### 保存结果

点击一次就新建一篇知识笔记，不覆盖原有笔记。

如果当前运行环境可用本地项目目录和知识目录：

- 笔记写入知识库
- 同时写入对应的 `.md` 镜像文件

如果本地镜像路径不可用：

- 仍保存到知识库
- 不阻断主流程

## 数据与保存规则

### 可保存内容范围

保存内容只取 assistant 消息的正文 Markdown：

- 不包含 `thinking` 块内容
- 不包含 `tool` 卡片内容
- 不包含保存状态提示文案
- 不包含知识提案卡片内容

### 标题规则

标题按以下顺序生成：

1. 如果正文里存在第一条有效 Markdown H1（`# 标题`），使用该标题
2. 否则使用正文第一行可读文本
3. 如果第一行过长，按现有风格截断到合理长度
4. 如果仍无法得到稳定标题，回退为 `会话标题 + 时间戳`

### 内容标准化

保存前使用现有 `serializeKnowledgeNoteMarkdown` 生成规范化 Markdown：

- 标题和正文格式与知识笔记保持一致
- 保存后文档被当作普通知识 Markdown 文档处理
- 不额外添加 AI 来源说明
- 不自动追加来源标签

### 文件与去重

每次保存都调用现有“新建知识笔记”链路：

- 创建新的知识笔记记录
- 如可用，则通过现有镜像路径生成逻辑写入 `.md` 文件
- 如果标题重名，沿用现有 suffix 机制生成不同文件名
- 不执行更新已有笔记的逻辑

## 实现范围

### UI 层

在聊天消息列表里增加普通 assistant 消息的保存操作区：

- `保存到项目`
- `保存中...`
- `已保存到项目`
- 错误提示

正文渲染切换为 `KnowledgeMarkdownViewer`，并保持现有消息气泡、时间、错误态和提案卡片布局。

### 状态层

需要为单条消息补一个轻量保存状态，建议为 UI 本地状态或消息级派生状态，范围仅限聊天视图：

- `idle`
- `saving`
- `saved`
- `error`

这次不把保存状态写入长期项目数据模型，也不把它设计成新的全局 store。

### 能力复用

直接复用现有能力：

- `KnowledgeMarkdownViewer`
- `serializeKnowledgeNoteMarkdown`
- `createProjectNote`
- 本地知识目录镜像写入逻辑

不新增第二套“项目 Markdown 文档保存器”。

## 测试策略

第一版只补最关键的回归验证：

1. 聊天源码测试
   验证聊天已引入 `KnowledgeMarkdownViewer`，并包含 `保存到项目`、`保存中...`、`已保存到项目` 等文案或逻辑。

2. 保存链路测试
   验证普通 assistant 消息保存时走 `serializeKnowledgeNoteMarkdown`、`createProjectNote` 和镜像写入逻辑。

3. 行为边界测试
   验证保存逻辑是“新建”而不是“更新”，且不会复用当前聚焦知识笔记。

4. 消息结构回归
   验证 `thinking` / `tool` 解析链路仍然存在，只有正文展示方式切到 Markdown。

## 风险与约束

- 现有 `KnowledgeMarkdownViewer` 的样式是为知识阅读场景设计的，接入聊天后需要轻量适配消息气泡内的间距与层级。
- 聊天回答可能不是严格规范的 Markdown，因此标题提取必须容错，不能因为缺少 H1 导致保存失败。
- 如果一条回答同时含有大量结构块与正文，保存逻辑必须明确只提取正文，避免把思考或工具内容误存为正式文档。

## 验收标准

- assistant 普通回答可以在聊天中按 Markdown 正常阅读。
- 聊天里的思考块和工具块行为不变。
- 每条可保存的 assistant 回答都可以一键保存到项目。
- 保存结果是新的知识库笔记，而不是覆盖已有笔记。
- 本地知识目录可用时，会同步生成 `.md` 镜像文件。
- 保存后的文档没有额外的 AI 来源标签或 AI 身份说明。
