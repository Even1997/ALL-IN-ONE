# Product Workbench 单一搜索与知识区头部收口设计

## 摘要

本次调整只做产品工作台现有壳层的收口，不改数据模型、不改文件树来源、不改 Markdown 编辑能力。目标是把左侧栏顶部的项目名称替换为单一搜索框，并移除知识库区和页面区内部重复的顶部搜索入口；同时在从左侧文件树打开 Markdown 文件时，精简右侧主内容区顶部的冗余文案和“新建”类按钮。

## Goals

- 左侧栏顶部不再展示项目名称，改为一个统一搜索框。
- 统一搜索框在“知识库”页签下搜索知识树，在“页面”页签下搜索页面树。
- 移除知识库区内部的 `搜索文档` 输入框。
- 移除页面区内部的 `搜索页面` 输入框。
- 从左侧文件树打开 Markdown 文件后，右侧主区不再展示文件类型 / 状态 / 更新时间这类顶部说明文案。
- 移除知识库主区顶部的“新建”按钮。

## Non-Goals

- 不新增全局跨页签搜索能力。
- 不改知识树和页面树的过滤逻辑，只改搜索入口位置。
- 不改右键菜单中的文件管理能力。
- 不移除上传、阅读 / 编辑、删除等仍然必要的操作。
- 不改 HTML 文件打开态的核心展示方式。

## 方案选择

采用“左上角单一搜索框，按当前页签工作”的方案。

原因：

- 改动最小，可以直接复用现有 `knowledgeSearch` 和 `pageSearch` 状态。
- 用户心智简单：当前看哪个页签，就搜索哪个树。
- 避免一个输入同时影响两套列表，减少不可见状态联动。

放弃“全局同时搜索知识库和页面”的原因：

- 输入行为会同时影响两个区域，但用户当下只能看到其中一个区域。
- 对这次收口目标来说，这是额外复杂度，不值得。

## 交互设计

### 左侧顶部

- 将 `pm-nav-header` 从静态项目名称改为搜索输入框。
- 占位文案根据当前页签切换：
  - 知识库：`搜索文档`
  - 页面：`搜索页面`
- 输入值直接绑定当前页签对应的搜索状态：
  - `sidebarTab === 'requirement'` 时读写 `knowledgeSearch`
  - `sidebarTab === 'page'` 时读写 `pageSearch`

### 左侧页签内部

- 知识库区保留条目计数与树结构，删除区块内部搜索框。
- 页面区保留页面计数与页面树，删除区块内部搜索框。

### 知识库主内容区

当打开的是 Markdown 文件时：

- 保留文件标题。
- 移除文件类型 / 状态 / 更新时间组合文案。
- 保留阅读 / 编辑切换、删除按钮。
- 若文件是 `sketch`，保留“用于设计”按钮。

当展示知识库主卡片头部时：

- 移除“新建”按钮。
- 保留“上传”按钮。

## 实现边界

- 主要改动集中在 `src/components/product/ProductWorkbench.tsx`。
- 如需微调搜索框或头部间距，仅补充 `src/App.css` 中已有样式，不新增无关样式体系。
- 不拆组件，不做无关重构。

## 验证

1. 在“知识库”页签输入关键字，只过滤知识树。
2. 切到“页面”页签输入关键字，只过滤页面树。
3. 两个页签内部原有搜索框都不再渲染。
4. 左侧顶部不再显示项目名称。
5. 打开 Markdown 文件后，右侧顶部不再显示类型 / 状态 / 时间说明。
6. 知识库主卡片头部不再显示“新建”按钮。

## 风险与约束

- 单一搜索框切换页签后会切换搜索语义，这是有意设计，不视为问题。
- 搜索状态仍分别保留；如果用户先后在两个页签输入不同关键字，切回时会恢复各自之前的过滤结果。
