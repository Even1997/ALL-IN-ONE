# Codex 风格消息输出流设计

- 日期：2026-05-04
- 状态：已确认，待实现计划
- 范围：仅调整 AI 聊天消息中的执行输出流展示，不改右侧 runtime 面板或整体工作台布局

## 背景

当前运行时事件已经能在聊天消息中展示工具调用、审批、提问和文件变更，但整体呈现仍偏技术调试视图：

- 执行步骤以 `details/summary` 卡片为主，默认展开逻辑更偏排查问题
- 工具输入、输出和 JSON 细节暴露较早，打断正文阅读
- 连续读取、搜索、编辑等动作虽然已分组，但分组标题和视觉层级仍不够产品化
- 深浅色主题下的步骤流视觉没有形成一套独立、稳定的设计语言

目标是把这部分体验改成更接近 Codex 的消息输出流：默认简洁、节奏清楚、可感知正在执行什么；需要排查时，再单步展开看到真实细节。

## 目标

1. 在 assistant 消息内部展示轻量执行步骤流，而不是把所有工具内容直接摊开。
2. 默认折叠技术细节，只显示短标签、状态和少量计数信息。
3. 单步展开后可查看真实执行内容，包括工具名、命令、文件路径、输出摘要和必要原始输出。
4. 保持现有 runtime 数据来源不变，优先改渲染结构和样式，不重写底层事件模型。
5. 同时适配浅色和深色主题，整体视觉风格更接近 Codex：克制、紧凑、层级清晰。

## 非目标

- 不改右侧 `GNAgent` runtime 侧栏、Timeline、Tools 面板的信息结构
- 不改聊天输入框、会话列表、页面整体布局
- 不新增新的运行时事件类型
- 不把原始终端或工具输出迁移到单独页面

## 已确认的产品决策

- 仅改“消息输出流”，不改整套右侧 AI 页面
- 风格参考 Codex
- 结构采用“轻量步骤流 + 单步展开真实细节”
- 信息密度偏真实，但默认折叠
- 步骤流内需要保留类似“分析中 / 读取文件 / 修改中 / 完成”的阶段感
- 展开后可以看到更完整的技术细节

## 方案对比

### 方案 A：全部展开的真实执行流

优点：

- 最接近原始 runtime 数据
- 实现改动较小

缺点：

- 消息流噪音高
- 正文可读性差
- 不符合目标中的“默认克制”

### 方案 B：单个总折叠卡片

优点：

- 最省空间
- 视觉最安静

缺点：

- 执行节奏感弱
- 不像 Codex 那种“边执行边可感知”的体验

### 方案 C：轻量步骤流 + 单步展开真实细节

优点：

- 默认简洁
- 保留执行节奏
- 需要时能查真实技术内容
- 与现有分组模型兼容

缺点：

- 需要重新组织步骤摘要与展开层级
- 需要补一套更细的主题样式

结论：采用方案 C。

## 信息架构

每条带 runtime 事件的 assistant 消息分成三层：

### 1. 正文层

正常展示 assistant 文本回复，不插入大段工具技术内容。正文优先，执行流辅助。

### 2. 轻量步骤流层

正文之后展示一条紧凑的步骤流，使用小体量节点或细卡片表达执行过程。每个步骤默认只显示：

- 状态
- 短标签
- 必要的数量信息

示例：

- 分析中
- 读取 3 个文件
- 搜索代码 2 次
- 编辑 1 个文件
- 完成

### 3. 单步展开层

点击某一步时，仅展开当前步骤，展示该步骤的真实细节：

- 工具名
- 请求摘要或命令
- 文件路径
- 文件变更摘要
- 输出预览
- 必要时的原始输出或技术细节

默认同一时间仅展开一个步骤，避免消息流无限拉长。

## 组件与职责

### `AIChatRuntimeToolExecutionCard`

职责：

- 作为消息内 runtime 输出流总容器
- 渲染“步骤流 + 单步展开”的整体框架
- 负责汇总统计，例如步骤数、完成态、失败态

调整方向：

- 保留现有 runtime 事件入口
- 从“树状工具卡片列表”调整为“轻量步骤流容器”

### `AIChatRuntimeToolBlocks`

职责：

- 渲染单个步骤、步骤组、独立结果块

调整方向：

- `RuntimeToolTree` 变成轻量步骤节点
- `RuntimeToolGroup` 变成更产品化的阶段节点，如“读取 3 个文件”
- 技术细节继续下沉到展开区，而不是首屏显示
- 如果某步没有可展开细节，不显示展开 affordance

### `runtimeEventRenderModel`

职责：

- 保持当前对 runtime events 的整理与分组

调整方向：

- 延续现有批量读操作分组逻辑
- 把分组文案从技术描述改成更面向用户的短标签
- 明确哪些事件进入步骤流，哪些事件作为步骤附属内容展示

### 样式层 `chat-tool-trace-*`

职责：

- 提供消息流内的步骤节点、展开区、状态色和深浅色主题表现

调整方向：

- 从“调试卡片”改为“低干扰步骤脊柱”
- 默认缩小视觉重量
- 强化运行中、失败、等待审批三类状态
- 深浅色分别定义 token，但共用同一层级逻辑

## 数据映射规则

### 步骤标签

优先使用现有 helper 能力生成 headline 和 summary，但调整文案输出方向：

- 连续读文件操作：`读取 N 个文件`
- 连续搜索：`搜索代码 N 次`
- 连续编辑：`编辑 N 个文件`
- 审批挂起：`等待批准`
- 用户提问：`等待输入`
- 独立结果：`完成`、`失败`、`已拦截`

### 步骤状态

保留现有状态集合：

- `running`
- `completed`
- `failed`
- `blocked`

并通过视觉映射为：

- 运行中：轻高亮
- 已完成：低对比
- 失败：明显强调
- 阻止/等待：中等强调，避免与失败混淆

### 展开内容

展开时优先展示：

1. 请求摘要
2. 文件变更
3. 输出摘要
4. 技术细节

技术细节区仍可保留二级折叠，用于 JSON 输入和较长原始输出。

## 交互规则

1. 默认折叠所有步骤细节。
2. 当前运行步骤可根据现有 helper 规则默认展开，或仅高亮但不完全展开。实现阶段应选择更克制的默认行为。
3. 点击某一步后，收起其他已展开步骤，只保留一个展开项。
4. 无细节的步骤只显示静态状态，不提供展开箭头。
5. 审批和提问事件继续作为步骤附属块展示，但视觉上从属于当前步骤。

## 视觉设计

### 整体气质

- 更像 Codex，而不是 IDE 调试面板
- 细、紧凑、连续
- 先看懂过程，再查技术细节

### 深色主题

- 基础背景偏石墨黑与蓝灰
- 当前步骤使用低强度冷色边框或微光
- 展开区采用更深层级背景，形成信息下沉

### 浅色主题

- 基础背景偏冷白或暖灰白
- 默认边框弱化
- 当前步骤通过轻底色和细描边突出

### 状态表达

- 运行中：细微动态或亮边
- 已完成：低对比中性色
- 失败：更高对比但不过度刺眼
- 等待输入/审批：明确但不等同失败

## 错误处理

- 如果某条消息只有零散 `tool_result`，继续走独立结果块渲染，但视觉上也纳入同一套步骤流
- 如果某步缺失 result，仍按 `tool_use` 信息生成步骤，状态使用当前已知值
- 如果技术输出为空，展开层只展示已有摘要和文件变更
- 如果某些旧数据没有 parent-child 关系，保持回退树结构，但沿用新样式

## 测试策略

### 组件层

- 验证步骤默认折叠
- 验证单步展开行为
- 验证无细节步骤不显示展开 affordance
- 验证连续读/搜/改操作的分组文案

### 运行时场景

- 仅工具调用
- 工具调用 + 文件变更
- 工具调用 + 审批
- 工具调用 + 提问
- 失败步骤
- 独立结果事件

### 主题验证

- 深色主题可读性
- 浅色主题可读性
- 长路径和长输出内容换行表现

## 成功标准

- assistant 消息中的执行输出在默认状态下明显更简洁
- 用户可以快速看懂当前做了什么，而不是先看到技术细节
- 展开某一步后能看到真实工具内容，信息不丢失
- 深色和浅色主题都成立
- 改动仅集中在消息流渲染和样式层，不牵连右侧 runtime 面板
