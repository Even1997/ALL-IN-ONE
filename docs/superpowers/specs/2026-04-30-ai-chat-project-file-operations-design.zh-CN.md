# AI Chat 项目文件增删改查设计

## 摘要

这个设计为现有 AI 聊天补上一套面向项目目录的文件操作能力，让 AI 可以在聊天中直接完成：

- 查询和读取项目文件
- 新建项目文本文件
- 编辑项目文本文件
- 删除项目文本文件

第一版遵循“读操作直出，写操作受模式约束”的原则：

- `查询 / 读取` 直接执行，不需要用户确认
- `新建 / 编辑 / 删除` 由本次聊天模式控制
  - `手动确认`：先提示，再由用户确认执行
  - `自动确认`：系统直接执行，不等待用户点击确认

这次能力面向“当前项目根目录内的任意文本文件”，不把二进制写改删纳入第一版主流程。

## 目标

- 让 AI 聊天具备对当前项目文本文件的可控 CRUD 能力。
- 保持聊天体验连续，不要求用户切出到文件管理器才能让 AI 改文档。
- 为高风险操作提供明确的模式切换，而不是一刀切阻止。
- 复用现有项目文件工具链，避免再造一套独立的文件系统层。
- 所有执行结果都能在聊天中回显，避免黑箱修改。

## 非目标

- 不让 AI 在第一版中写改删图片、PDF、音视频或其他二进制文件。
- 不开放递归删除整个目录。
- 不允许操作跳出当前项目根目录。
- 不把这次设计扩展成完整 IDE 或通用 shell。
- 不在第一版加入复杂权限系统、角色系统或多级审批。

## 当前问题

现有聊天能力已经可以：

- 基于项目上下文回答问题
- 读取部分知识和生成产物
- 在特定知识提案链路里保存文档

但它还缺少一套通用的“项目文件工作能力”：

- 用户不能自然地说“帮我新建一个 PRD 文件”
- 用户不能直接让 AI 编辑某个现有 Markdown 或 JSON 文件
- 用户不能让 AI 删除某个废弃文本文件
- 读、写、删的边界和确认方式没有统一模型

结果是很多文档类和项目文件类工作，用户仍要手动切回文件系统或编辑器自己处理。

## 设计原则

- 查询与读取保持轻量，直接做，不增加无意义确认。
- 写操作必须可解释：系统要知道准备改什么，而不是只给出一句“我改好了”。
- 模式切换影响“是否等待用户确认”，不影响执行结果回显。
- 项目目录边界必须是硬约束，不能依赖模型自觉。
- 第一版只做文本文件，范围收紧优先于过早追求“真正任意文件”。

## 用户体验

### 聊天模式切换

在聊天框下方增加“本次聊天模式”选择：

- `手动确认`
- `自动确认`

这是会话内的交互状态，用于控制本轮聊天中的写操作执行方式。

模式语义：

- `手动确认`
  - `查询 / 读取` 直接执行
  - `新建 / 编辑 / 删除` 先生成操作提示卡片，等待用户确认
- `自动确认`
  - `查询 / 读取` 直接执行
  - `新建 / 编辑 / 删除` 系统直接执行，不等待用户点击确认

### 查询与读取

以下能力可以直接执行并把结果回显到聊天：

- 查看目录
- 搜索文件
- 读取文本文件内容
- 查看某个路径是否存在

这些能力本质上属于低风险只读操作，不进入确认流。

### 写操作提示

当 AI 判断需要进行 `新建 / 编辑 / 删除` 时，系统应先生成结构化操作项，再决定是否执行。

在 `手动确认` 模式下：

- 聊天里显示一张“文件操作卡片”
- 展示即将执行的操作列表
- 用户点击确认后才真正落盘

在 `自动确认` 模式下：

- 系统不等待用户点击确认
- 直接执行操作
- 聊天中仍然展示执行摘要和结果

### 结果回显

无论哪种模式，执行后都要在聊天里显示：

- 操作类型
- 目标路径
- 成功或失败
- 简短改动摘要

编辑文件时，第一版应尽量展示简短 diff 或替换摘要；如果底层能力不足，至少展示“编辑了哪个文件、改动了什么类型的内容”。

## 支持范围

### 第一版正式支持的文件类型

第一版正式支持以下文本文件的 `新建 / 编辑 / 删除 / 读取`：

- `.md`
- `.markdown`
- `.txt`
- `.json`
- `.js`
- `.jsx`
- `.ts`
- `.tsx`
- `.css`
- `.html`
- `.yml`
- `.yaml`

如果现有底层能力允许，也可以兼容更多纯文本后缀，但不把它们写死为第一版验收要求。

### 二进制文件

二进制文件不属于第一版主流程。

允许的最低限度行为：

- 如果系统有办法读取元信息或预览文本化片段，可以作为“尽力读取”补充

第一版不支持：

- 写入二进制文件
- 编辑二进制文件
- 删除二进制文件

## 操作模型

### 统一的写操作类型

第一版建议统一成三种结构化操作：

- `create_file`
- `edit_file`
- `delete_file`

这样无论最终是手动确认还是自动确认，都走同一条执行链路。

### 新建文件

新建文件操作应至少包含：

- 目标路径
- 文件内容
- 操作摘要

如果目标路径已存在：

- 在 `手动确认` 模式下，必须明确提示这会变成覆盖风险，不能静默当作新建成功
- 在 `自动确认` 模式下，也不能静默覆盖，必须把它转为高风险编辑或报错

### 编辑文件

第一版优先支持两种编辑方式：

- 全量重写文件内容
- 基于旧字符串到新字符串的替换

推荐优先复用已有文本写入和局部替换能力，而不是在第一版引入复杂 patch 语法。

### 删除文件

第一版只支持删除单个文件，不支持：

- 删除整个目录
- 递归删除目录树

删除永远属于高风险写操作，在 `手动确认` 模式下必须等待确认，在 `自动确认` 模式下可以直接执行，但仍需受系统硬边界约束。

## 项目边界与安全约束

### 根目录边界

所有文件操作都必须限制在当前项目根目录内。

系统在执行前必须做路径规范化和边界校验：

- 不允许 `..` 越界
- 不允许通过绝对路径跳出项目
- 不允许借助路径拼接绕过根目录约束

### 文本文件约束

只有被判定为文本文件的目标，才允许进入 `新建 / 编辑 / 删除` 流程。

如果目标不是支持的文本文件：

- 读取可尽力而为
- 写改删直接拒绝并回显原因

### 自动确认模式的硬保护

`自动确认` 只是“无需用户再点确认”，不是“什么都能自动做”。

即使处于 `自动确认` 模式，系统仍然必须拒绝：

- 越出项目根目录的路径
- 递归删除目录
- 对非文本文件进行写改删

## 架构建议

### 推荐路径：结构化操作提案执行器

推荐把这套能力做成“AI 先产出结构化意图，前端执行器再决定是否执行”的架构。

原因：

- 读操作仍可快速直出
- 写操作可以统一走模式判断
- 更容易做回显、测试和安全校验
- 比把文件工具完全裸露给模型更可控

### 不推荐第一版直接裸露所有写工具

如果直接让聊天运行时把 `write / edit / remove` 完全裸露给模型：

- 手动确认模式会很难插入
- 自动确认和回显会更像黑箱
- 删除和覆盖风险更难审计

因此第一版应避免“模型边想边直接落盘”的默认路径。

## 与现有能力的关系

### 读取能力

现有目录查看、文件读取、文本查看能力可以继续复用，仍作为低风险只读能力使用。

### 写能力

现有文本写入、文本替换、文件删除底层能力可以作为执行器后端复用，但它们不应直接决定聊天交互。

聊天层负责：

- 识别这是读操作还是写操作
- 根据模式决定是否等待用户确认
- 组织回显结果

执行器负责：

- 校验路径和类型
- 真正调用底层文件能力
- 回传执行结果

## 测试策略

第一版测试应覆盖以下几类：

1. 模式测试
   - `手动确认` 下写操作只生成提示，不直接执行
   - `自动确认` 下写操作直接执行

2. 边界测试
   - 越出项目根目录的路径被拒绝
   - 目录删除被拒绝
   - 非文本文件写改删被拒绝

3. 读操作测试
   - 目录查看和文件读取不会进入确认流

4. 执行结果测试
   - 新建、编辑、删除后聊天里有正确回显

5. 回归测试
   - 现有 Markdown 阅读和知识提案链路不被破坏

## 风险与约束

- 如果第一版同时支持太多编辑语义，AI 侧输出和执行器侧约定会变复杂，因此应先收敛到“新建 / 全量重写 / 字符串替换 / 单文件删除”。
- 自动确认模式虽然更顺滑，但如果没有强路径边界和类型边界，很容易越权改错文件，因此系统约束必须在执行前生效。
- 读取二进制文件的“尽力支持”不能反向拉大第一版范围，不应演变成支持编辑二进制。

## 验收标准

- 聊天框下方存在本次聊天模式切换：`手动确认 / 自动确认`
- `查询 / 读取` 在两种模式下都直接执行
- `新建 / 编辑 / 删除` 在 `手动确认` 下必须等待用户确认
- `新建 / 编辑 / 删除` 在 `自动确认` 下无需用户点击即可执行
- 所有写操作都限制在当前项目根目录内
- 第一版只允许对文本文件执行写改删
- 删除目录和越界路径会被拒绝
- 所有执行结果都会在聊天中回显
