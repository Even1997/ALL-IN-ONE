# GoodNight 技能库页面视觉与交互重构设计

## 摘要

本设计在现有“全局技能库”架构不变的前提下，把当前偏工具面板式的 `Global Skills` 页面升级为一个更像产品首页的技能库页面。目标效果参考你给的 Codex 技能页截图：有明确的大标题、搜索入口、分类筛选、推荐区、系统区、个人区，以及一眼可识别的安装状态。

这次设计只重构技能页的视觉层与交互层，不推翻已有的 GoodNight skill registry、导入、删除这些后端边界；同时明确把“技能库”和“Claude / Codex 运行时”拆成两个独立概念。

## 目标

- 让用户进入技能页后先“浏览和理解”，再决定安装、导入或删除。
- 让 `内置技能`、`推荐技能`、`个人技能` 的差异一眼可见。
- 让已安装、可导入、可更新等状态在卡片层就能快速识别。
- 保持聊天窗口仍然是普通聊天，只通过 `@skill` 调用技能。
- 在不扩大后端复杂度的前提下，让技能页更接近一个真正可用的“技能库”而不是调试面板。

## 非目标

- 不在第一版引入公开技能市场、评分、评论、排行榜。
- 不引入账号体系、付费安装或云端同步。
- 不把 `插件` 与 `技能` 在第一版做成完整双生态。
- 不新增复杂审核流、多人协作流、权限市场。
- 不在技能首页混入 `Claude`、`Codex`、`Built-in AI` 这类运行时概念。

## 当前问题

当前 `GNAgentSkillsPage` 已经具备最小功能，但从产品体验上仍有明显问题：

- 页面更像操作面板，不像技能库首页。
- 用户进入页面后先看到按钮，再看到技能本身，浏览体验很弱。
- 当前页面把技能来源和运行时能力混在一起，用户很难判断“这是一个技能”还是“这是某个代理/运行时的配置项”。
- 本地导入和 GitHub 导入依赖 `prompt`，流程更像开发者工具，不像桌面产品。
- 缺少搜索、筛选和分组，技能数量一多后会迅速失控。
- 运行时相关动作直接出现在技能卡主层，会抢走“这个技能是什么、值不值得装”的认知层。

## 设计原则

### 先看库，再做动作

技能页首先应该是一个可浏览、可理解、可筛选的库，其次才是操作入口。

### 分类必须稳定

第一眼就要区分三类内容：

- `推荐`
- `系统`
- `个人`

不要让所有来源混成一个平铺列表。

### 已安装状态优先于技术来源

对普通用户来说，“有没有装、能不能用”比“它来自哪个目录”更重要。来源信息应保留，但视觉优先级低于安装状态。

### 技能与运行时必须分层

`技能` 是能力单元，`Claude / Codex / Built-in AI` 是运行时或代理入口，它们不是同一层对象。

因此：

- 技能页只管理技能本身
- 运行时配置放到独立的 `Agents`、`Runtimes` 或设置页
- 不在技能首页把技能卡直接做成“同步到某个代理”的控制面板

### 保持现有后端边界

前端可以更产品化，但不应该因此把技能架构改成完全不同的模型。第一版尽量复用现有 discovery、import、delete 命令；与运行时有关的能力即使底层仍存在，也不应成为技能首页主叙事。

### 聊天与管理继续分离

技能管理页负责管理技能；聊天页负责使用技能。不要再把技能库面板塞回聊天窗口。

## 页面结构

### 整体布局

页面采用和截图相近的“内容首页”结构：

- 顶部轻操作栏
- 中间标题区
- 搜索与筛选区
- 分组内容区

桌面端优先双列卡片网格，窄屏退化为单列。

### 顶部轻操作栏

页面右上角保留轻量操作：

- `管理`
- `导入`
- `刷新`

如果后续真的要做“创建技能”，可以作为 `导入` 旁边的扩展动作，但不作为这次设计的硬要求。

这里不建议直接照搬截图里的 `插件 / 技能` 顶部双标签，因为 GoodNight 当前第一版要解决的是 `Skills` 页面，不是同时落地一个完整插件市场。

### 标题区

标题区应明显强于当前实现，建议采用一句偏产品化的标题，例如：

- `让 GoodNight 按你的方式工作`

副标题可短句说明：

- 在这里浏览、导入和管理全局技能；在聊天里通过 `@skill` 调用它们。

这一区域的作用不是解释全部架构，而是给页面一个清晰身份，让它看起来像“技能库首页”而不是技术设置页。

### 搜索与筛选区

标题下方提供统一搜索栏和一个分类/状态筛选器。

### 搜索

搜索至少覆盖：

- 技能名
- `id`
- 描述
- 分类标签

默认占位文案可为：

- `搜索技能`

### 筛选器

第一版建议保持一个简单下拉，而不是做复杂多选面板。可选项：

- `全部`
- `已安装`
- `未安装`
- `推荐`
- `系统`
- `个人`
- `可更新`

如果数据暂时不够支撑全部选项，也至少保留：

- `全部`
- `已安装`
- `推荐`

### 分组内容区

技能库主内容分为三个稳定分组。

### 推荐

这一组展示 GoodNight 希望用户优先关注的技能。它们可以来自：

- GoodNight 官方推荐目录
- 已知优质 GitHub 技能
- 适合作为开箱即用能力的精选内置技能

卡片右侧动作根据状态切换：

- 未安装：`+`
- 已安装：对勾状态
- 可更新：更新提示

推荐区的重点不是展示全部元数据，而是快速回答：

- 这是什么
- 适合拿来做什么
- 我能不能一键纳入技能库

### 系统

这一组展示 GoodNight 自带或平台级基础技能，例如：

- 边界约束型技能
- 工作区上下文技能
- 默认安全守护技能

系统技能默认已安装，可见但通常不可删除。它们的视觉状态应更稳定，避免让用户误以为这是随手可移除的外部能力。

系统区卡片动作建议收敛为：

- 已安装状态
- 查看详情

不在这一组展示删除入口。

### 个人

这一组展示真正由用户自己管理的技能集合，包括：

- 本地手动导入技能
- GitHub 下载后纳入 GoodNight 的技能
- 从外部目录发现并纳入 GoodNight 的技能

个人区是最接近“我的技能”的区域，因此允许更多管理动作：

- 导入
- 删除

如果某个技能只是“已发现但未导入”，它仍应出现在个人区，但状态应清楚标记为：

- `可导入`

而不是伪装成已经安装。

## 技能卡设计

每张技能卡保持高度克制，不做大面积说明文。建议结构如下：

- 左侧图标或首字母占位
- 名称
- 一行短描述
- 次级元数据
- 右侧主状态 / 主动作

### 卡片信息层级

卡片至少显示：

- 技能名
- 简短描述
- 分类或来源标签
- 安装状态

在展开态、悬停态或次级区域中再显示：

- `id`
- 本地路径
- 来源目录
- 风险标记
- 更新时间或版本信息

不要默认把完整路径放成卡片最显眼的一行。当前实现把路径放得太靠前，更像文件浏览器而不像技能库。

### 卡片状态

第一版统一为以下几种视觉状态：

- `未安装`
- `已安装`
- `可导入`
- `可更新`

其中：

- `已安装` 应是最醒目的稳定正向状态
- `可导入` 是外部发现状态
- 运行时适配信息如果保留，也只能是详情层信息，不应压过安装状态

### 卡片动作

卡片右侧只保留一个主动作，其余动作下沉到次级按钮或更多菜单，避免三四个并列按钮把卡片挤满。

推荐映射：

- 未安装推荐技能：主动作 `安装`
- 已发现本地技能：主动作 `导入`
- 已安装技能：主状态 `已安装`

次级动作可以是：

- `删除`
- `查看详情`

这会比当前“每张卡横着摆三颗按钮”更接近截图里的产品感，也更符合“技能页不混运行时”的边界。

## 交互细节

### 导入本地技能

不建议继续长期依赖 `window.prompt` 输入路径。第一版至少应升级为轻量对话框或表单卡片，包含：

- 路径输入
- 路径粘贴
- 确认导入

如果当前桌面能力暂时不足，短期可以保留底层命令不变，但交互层应明确朝可视化表单过渡。

### 从 GitHub 下载

同样不建议让用户连续填写三次 `prompt`。建议改为一个小表单：

- `owner/repo`
- skill path
- 可选 ref

提交后在页面顶部给出成功或失败反馈。

### 查看详情

点击卡片或更多菜单后，可以打开右侧抽屉或轻量详情面板，展示：

- 技能描述全文
- 来源
- 本地路径
- 风险标记
- 版本与更新时间

详情不是这次设计的唯一核心，但它能避免把过多元数据硬塞进卡片正面。

## 数据映射建议

现有 `SkillDiscoveryEntry` 还不足以支撑截图式技能页。为了不推翻后端，建议在现有字段上做渐进增强。

### 现有可复用字段

- `id`
- `name`
- `source`
- `path`
- `imported`
- `builtin`
- `deletable`

### 建议补充字段

- `description`
- `category`
- `author`
- `icon`
- `recommended`
- `updateAvailable`

其中：

- `category` 用来稳定映射到 `推荐 / 系统 / 个人`
- `description` 优先从 manifest 读取；没有时可退化为 `SKILL.md` 第一段摘要
- `icon` 没有专门资源时可自动生成首字母图标
- `recommended` 可先由 GoodNight 内置 catalog 标记

### 第一版分组映射

即使暂时不补齐所有字段，也可以先用现有数据完成稳定分组：

- `builtin === true` → `系统`
- `imported === true && builtin === false` → `个人`
- `source` 属于 curated catalog → `推荐`
- 本地发现但未导入 → `个人 / 可导入`

这样能在不重写架构的前提下先把页面层次做出来。

## 与现有架构的关系

本设计是对以下既有方向的补强，而不是替代：

- `2026-04-26-goodnight-unified-skill-library-and-runtime-sync-design.zh-CN.md`
- `2026-04-30-global-skill-library-page-design.zh-CN.md`

前两份文档解决的是：

- 技能真源在哪里
- 如何导入与落库
- 为什么技能页要和聊天分离

这次补的是：

- 技能页长什么样
- 技能卡怎么排
- 用户如何更自然地浏览、筛选、安装和管理技能

## 实现建议

### 第一阶段：只重构页面信息架构

在不改变后端命令的前提下先完成：

- 标题区
- 搜索栏
- 简单筛选器
- `推荐 / 系统 / 个人` 三分组
- 卡片主动作收敛
- 把运行时相关表达从技能首页主层移走

### 第二阶段：补元数据

按需补：

- description
- recommended
- support matrix
- updateAvailable

让卡片真正从“文件条目”变成“技能条目”。

### 第三阶段：替换 prompt 式导入交互

把：

- 本地导入
- GitHub 下载

从浏览器 `prompt` 升级为页面内表单或桌面对话框。

## 测试策略

第一版至少覆盖以下几类验证：

1. 页面结构测试
   - 存在搜索栏
   - 存在 `推荐 / 系统 / 个人` 分组
   - 技能卡显示主状态而不是只显示路径

2. 状态映射测试
   - built-in 技能进入 `系统`
   - imported 非 built-in 技能进入 `个人`
   - 推荐 catalog 技能进入 `推荐`

3. 动作门控测试
   - 内置技能不可删除
   - 已安装技能显示安装状态
   - 技能卡主层不出现 `Claude / Codex` 运行时动作

4. 交互回归测试
   - 聊天页仍通过 `@skill` 使用技能
   - 技能管理仍停留在独立 `Skills` 页面

## 风险与约束

- 如果没有 `description`、`category` 之类的补充字段，页面容易只是“换皮”，而不是真正提升可浏览性。
- 如果继续长期依赖 `prompt` 式导入，页面即使视觉上更像技能库，核心使用体验仍会跳回开发者工具感。
- 如果继续把 `Claude / Codex` 放进技能首页主层，用户会把“技能”误解成“代理模式”或“运行时设置”。
- 如果第一版就同时做“插件 + 技能”双体系，容易把范围迅速拉大，反而延迟技能页落地。

## 验收标准

- 技能页进入后首先看到的是标题、搜索和分组，而不是一排技术动作按钮。
- 页面稳定分为 `推荐 / 系统 / 个人` 三个区域。
- 用户能一眼区分“未安装”“已安装”“可导入”“可更新”等状态。
- 内置技能可见、稳定、不可删除。
- 个人技能可导入、可删除。
- 技能首页主层不出现 `Claude`、`Codex` 这类运行时控制。
- 聊天窗口继续保持普通聊天，不再承载技能管理面板。
- 页面整体气质更接近一个桌面产品的技能库首页，而不是调试工具面板。

## 结论

GoodNight 已经有了“全局技能库”的正确架构方向，但当前页面表达还停留在功能验证层。本次设计建议不动后端真源与导入边界，先把技能页升级成一个真正可浏览、可筛选、可理解的技能库首页，并明确把技能概念与 `Claude / Codex` 运行时拆开。

这样做的好处是：

- 不推翻已完成的技能库基础设施
- 能快速把体验拉到接近截图那种成熟产品状态
- 继续保持“聊天负责使用，技能页负责管理”的产品边界
- 避免把“技能”、“代理”、“运行时”混成一个页面概念
