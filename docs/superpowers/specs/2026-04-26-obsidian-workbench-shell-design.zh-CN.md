# Obsidian 风格工作台外壳与知识库整合设计

## 摘要

本设计将当前产品工作台重构为更接近桌面工具的统一外壳，并将知识区升级为接近 Obsidian 的本地知识库体验。新的工作台采用左侧导航、中间主工作区、右侧 AI 状态区的三栏结构，整体视觉遵循已有 `workbench` monochrome 方向：更克制的黑白灰表面、更紧的圆角、更少的漂浮感、更强的结构层级。

这次改造包含两条并行目标：

- 把产品工作台从“页面堆叠的功能区”提升为真正的桌面工作台外壳。
- 把知识区从普通详情视图升级为基于真实 Markdown 文件的 Obsidian 风格知识工作区。

本设计明确要求：虽然选择了整壳重做，但画布功能必须被视为受保护子系统。画布的数据结构、交互行为、持久化格式、草图文件规则以及 AI 对画布引用的协议都不能在这次改造中被改变。

## 目标

- 将当前工作台统一为左中右三栏的桌面应用布局。
- 在不引入底部终端/结果面板的前提下，让工作台结构完整成立。
- 将知识库体验升级为 Obsidian 风格：真实文件树、多标签、所见即所得 Markdown 编辑、本地全文搜索。
- 保持知识库以真实 Markdown 文件为 source of truth，不引入新的知识库后端服务。
- 让知识区、画布区、AI 区都落入同一套 monochrome workbench 视觉系统。
- 保证画布功能、草图页文件和设计板持久化链路不受这次改造影响。

## 非目标

- 不做底部终端/结果面板。
- 不做 Notion 式 block database。
- 不做插件系统、协作、云同步或知识图谱。
- 不引入向量数据库、RAG 服务或独立知识库后端。
- 不重写画布内核，不顺手调整画布节点模型、交互语义或落盘格式。
- 不改变 AI 工作流内部阶段定义，只调整它们在 UI 中的承载方式和上下文装配方式。

## 总体产品方向

这次改造后的产品应更像 `Obsidian x VS Code` 的混合体，而不是网页风格的 dashboard。它需要具备桌面工具应有的结构感：

- 顶部是更薄、更平、更少装饰的全局工具栏。
- 左侧是文件与结构导航，不再像浮动卡片列表。
- 中间是唯一的主工作面，知识编辑和画布工作都在这里发生。
- 右侧是固定的 AI 状态与上下文区，而不是长聊天窗口。

界面重心应从“卡片”转向“面板”，从“色块强调”转向“边界与层级强调”。

## 工作台外壳

### 三栏骨架

新的工作台骨架固定为：

- 左侧：导航区
- 中间：主工作区
- 右侧：AI 状态区

这一结构在桌面宽度下稳定存在，不依赖底部面板成立。

建议宽度策略：

- 左侧导航：`240px` 到 `280px`
- 右侧 AI 区：`360px` 到 `420px`
- 中间主工作区：占据剩余空间并始终保持主导地位

所有桌面分栏继续使用 `Allotment` 管理分割与尺寸记忆，沿用已有的全局布局记忆方案。

### 顶部工具栏

顶部工具栏应保留，但视觉上明显收紧：

- 高度更薄
- 去掉明显的发光与玻璃感
- 只保留全局级别操作，例如项目切换、视图切换、少量高优先级命令
- 不再承载过多局部信息

顶部工具栏是工作台 chrome 的一部分，不应再像一个独立漂浮的标题卡。

## 信息架构

### 左侧导航区

左侧导航区是统一导航外壳，而不是只服务某一个子页面。它在不同工作态下承载不同内容，但视觉和结构一致。

在知识工作态中，左侧主要负责：

- 知识树
- 文件搜索入口
- 分组与筛选
- 新建文件和文件夹等基础文件操作

在画布工作态中，左侧主要负责：

- 页面树或结构树
- 与画布相关的辅助导航

无论哪种工作态，左侧导航区都应保持同样的容器语言、同样的边界处理、同样的密度标准。

### 中间主工作区

中间主工作区是新的工作台核心，始终只承载一个当前主任务。

它在概念上分为两个并列工作态：

- `KnowledgeWorkspace`
- `PageWorkspace`

这两者属于同一个工作台，而不是两个分裂的产品。切换时，左侧与右侧壳体尽量保持稳定，只替换中间核心内容。

### 右侧 AI 状态区

右侧 AI 区固定存在，承担以下职责：

- 展示当前上下文范围
- 展示当前 AI 状态、最近一次结果摘要、下一步建议
- 提供引用范围选择
- 展示与当前知识页或当前画布页相关的状态卡片

右侧 AI 区不是传统聊天窗口，不应演变成长消息历史。它更接近“上下文与状态控制台”。

## 知识工作区

### 核心方向

知识区第一版明确采用 Obsidian 风格路线：

- 文件优先
- Markdown 优先
- 本地优先
- 写作与知识整理优先

它不追求 Notion 的 block database，也不追求 IDE 式源码编辑器体验，而是要在真实文件模型上提供更接近日常知识工作的操作感。

### 核心能力

第一版知识工作区必须支持：

- 真实文件树浏览
- 真实文件夹与 Markdown 文件的新建、重命名、删除、移动
- 打开多个知识文档标签
- 所见即所得 Markdown 编辑
- 自动保存到真实 `.md` 文件
- 标题与正文的本地全文搜索
- AI 引用 `当前文档 / 当前目录 / 已打开文档 / 全部知识`

### 第一版明确不做

第一版知识工作区不做：

- block database
- 双向链接图谱
- 数据库表格视图
- 协同编辑
- 云同步
- 插件安装体系

这样可以保证第一版专注于“更像 Obsidian 的本地知识库”，而不是把项目拖入一个新的产品方向。

### 编辑体验

知识区的编辑体验选择所见即所得，而不是纯源码 textarea。

这并不意味着放弃 Markdown 文件，而是要求：

- 用户看到的是富文本化、可直接编辑的 Markdown 内容
- 底层保存的仍然是真实 Markdown
- 用户不需要先理解源码模式才能进行日常编辑

如果后续需要再补源码模式，也应作为后续增强，而不是第一版范围。

### 技术选型

知识工作区第一版选型如下：

- 编辑器：`Milkdown`
- 搜索：`FlexSearch`
- 文件真相源：继续使用现有 `Tauri` 文件系统持久化
- 分栏布局：继续使用 `Allotment`

选择 `Milkdown` 的原因：

- 它比 block editor 更贴合 Markdown-first 的知识库模型
- 比自己从 `Tiptap` 拼装完整 Markdown 体验更快
- 比 `BlockNote` 更少引入 block 数据结构与 Markdown 文件之间的转换负担

选择 `FlexSearch` 的原因：

- 本地全文搜索接入快
- 足够支撑桌面知识库第一版
- 不需要把项目引向服务端检索架构

## 画布工作区保护边界

### 保护原则

虽然这次选择整壳重做，但画布必须被视为“受保护子系统”。知识区和外壳允许大改，画布内核不允许借机重写。

### 禁止改动的内容

本次改造中，不允许改动以下内容：

- 画布节点数据结构
- 设计板持久化格式
- 草图页文件生成规则
- 页面与 wireframe 的落盘路径约定
- AI 读取设计页、草图页、style 引用文件的协议
- 现有画布交互语义，例如选中、拖拽、缩放、连线、页面切换

### 允许改动的内容

本次改造中，允许改动以下内容：

- 画布外层容器
- 画布区域的留白、边框、背景和面板样式
- 画布与左侧导航、右侧 AI 区并排时的整体布局
- 非核心控件的视觉表现
- 标签栏、工具栏、状态标签的外观

换句话说，这次可以重做的是 `shell` 和 `surface`，不能碰的是 `canvas engine`。

## 系统边界拆分

为了避免在单个超大组件中继续堆积职责，新的工作台结构应在实现上拆成清晰边界：

- `WorkbenchShell`
- `KnowledgeWorkspace`
- `PageWorkspace`
- `AIContextPane`

### WorkbenchShell

负责：

- 左中右三栏布局
- 全局工作台 chrome
- 主题与视觉容器
- 分栏尺寸与布局记忆接入

不负责：

- 知识编辑逻辑
- 画布内部逻辑
- AI 具体工作流执行

### KnowledgeWorkspace

负责：

- 知识树联动
- 打开标签
- Markdown 编辑
- 文件操作
- 搜索
- 当前知识上下文的暴露

### PageWorkspace

负责：

- 现有画布与页面工作流的继续承载
- 画布当前上下文的暴露

不负责：

- 改写现有画布数据模型
- 改写草图页文件协议

### AIContextPane

负责：

- 统一展示当前上下文
- AI 引用范围控制
- 状态摘要
- 下一步建议

不负责：

- 直接承载完整聊天式控制流
- 接管知识编辑器或画布编辑器的主交互地位

## AI 上下文边界

AI 区仍然保留现有“右侧状态/引用区”的定位，但它必须能统一理解两类主工作态：

- 当前正在编辑的知识文档
- 当前正在编辑的画布页或设计页

知识工作态下，AI 引用范围至少支持：

- 当前文档
- 当前目录
- 已打开文档
- 全部知识

画布工作态下，AI 继续沿用当前设计页、草图页、style 参考文件的上下文装配方式，不改变既有协议。

知识编辑器不能反向接管 AI 工作流。AI 的推进逻辑仍然属于现有工作流和上下文装配层，而不是 `Milkdown` 自身。

## 数据流

### 知识区

1. 左侧知识树从真实文件系统扫描结果构建。
2. 用户在中间 `KnowledgeWorkspace` 打开目标 Markdown 文件。
3. `Milkdown` 加载文件内容并提供所见即所得编辑。
4. 编辑结果直接保存回真实 `.md` 文件。
5. 文件变化触发本地搜索索引刷新。
6. 右侧 AI 区根据当前文档、当前目录、已打开标签或全部知识组装引用范围。

### 画布区

1. `PageWorkspace` 继续使用现有 `Canvas`、页面树、草图页、设计板状态。
2. 画布相关状态继续沿用当前 store 与持久化逻辑。
3. 右侧 AI 区继续读取当前画布上下文与设计引用文件。

### AI 区

1. AI 区根据当前工作态读取知识上下文或画布上下文。
2. AI 区展示状态摘要、引用范围和下一步动作。
3. AI 工作流继续由现有流程层驱动，而不是由知识编辑器本身驱动。

## 迁移顺序

### 1. 抽离 WorkbenchShell

先固定左中右三栏外壳，建立统一布局和视觉容器，但先不接新知识编辑器。

目标：

- 新外壳成立
- 桌面三栏结构稳定
- 不依赖底部面板

### 2. 拆分中间工作区

把当前产品工作台明确拆成：

- `KnowledgeWorkspace`
- `PageWorkspace`

目标：

- 画布仍可原样工作
- 知识工作区拥有独立接入编辑器的边界

### 3. 接入知识编辑器

把现有知识详情区替换为 `Milkdown`，并补齐：

- 多标签
- 自动保存
- 真实文件读写联动

### 4. 接入本地搜索

使用 `FlexSearch` 建立标题与正文搜索，先完成本地全文搜索，再把搜索结果与 AI 引用范围联动。

### 5. 统一视觉系统

最后把知识区、画布区、AI 区全部压入同一套 monochrome workbench token 中，只改视觉，不再扩展行为范围。

这个顺序的意义是先换壳、再换知识内核、最后收视觉，降低“同时修改多条链路”带来的误伤风险。

## 测试与验证

本次改造至少需要验证以下场景：

- 左中右工作台骨架在桌面宽度下稳定成立
- 不依赖底部面板也能完整使用工作台
- 知识区能真实打开 Markdown 文件
- Markdown 文件能编辑、保存、关闭、再次加载
- 多标签打开与切换稳定
- 本地搜索支持标题与正文
- AI 能按 `当前文档 / 当前目录 / 已打开文档 / 全部知识` 组装知识引用范围
- 画布行为与当前版本一致
- 画布文件和设计板状态仍可正常保存与读取
- dark / light 主题都落入同一套 monochrome workbench 系统

## 验收标准

- 工作台第一眼看上去不再像网页 dashboard，而是更接近成熟桌面工具。
- 左中右三栏骨架清晰，中间主工作区始终占据视觉主导。
- 知识区明显更像 Obsidian，而不是普通文档详情卡片。
- 用户可以把知识库当作真实本地 Markdown 工作区使用。
- AI 区是状态与上下文区，而不是传统长聊天记录。
- 画布功能、草图文件、设计板持久化与 AI 画布引用链路都不因这次改造而退化。

## 已确认决策

- 不做底部终端/结果面板。
- 工作台采用左中右三栏结构。
- 知识库走 Obsidian 风格路线，而不是 Notion 风格路线。
- 编辑体验优先采用所见即所得 Markdown 编辑。
- 知识库第一版使用 `Milkdown + FlexSearch + 真实 Markdown 文件`。
- 画布功能和交互不改，只统一视觉和外层外壳。
- 本次虽为整壳重做，但不得改变画布数据链路和引用协议。

## 实现边界

本设计的关键不是一次性重写所有内部逻辑，而是在清晰边界下重组工作台：

- 允许重做工作台外壳
- 允许重做知识区体验
- 允许统一视觉系统
- 不允许把画布和 AI 上下文链路拖进范围失控的重写

第一版的成功标准不是“做出一个更复杂的新产品”，而是“让当前产品第一次真正拥有一个成立的桌面工作台与本地知识库体验”。
