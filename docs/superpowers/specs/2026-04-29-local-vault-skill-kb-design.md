# GoodNight 本地 Vault + 技能化知识库设计

## 结论

GoodNight 的知识库产品应从“内置 wiki 功能”切换为“本地 Vault 底座 + 技能化检索/整理能力”。

用户真正拥有和维护的是一个本地文件夹，体验尽量贴近 Obsidian：

- 用户打开或创建一个本地文件夹作为 Vault
- 主界面默认展示这个文件夹的原始文件树
- 用户原始文件是事实源，AI 不直接改写
- AI 只能新增自己的产出文件
- 系统内部索引和技能缓存放在隐藏目录 `.goodnight/`
- 用户可见的 AI 产出放在可见目录 `_goodnight/outputs/`
- 项目能力保留，但项目本质上只是 Vault 的入口壳
- 默认检索方式使用 `M-Flow`
- `LLMWiki / RAG / M-Flow` 作为可插拔 skill 存在

## 产品目标

### 目标

1. 让用户以“本地文件夹”理解和使用知识库，而不是黑盒项目容器
2. 让 AI 能基于同一个 Vault 使用不同技能做检索、整理、问答和文档生成
3. 区分用户原始文件、用户可见 AI 产出、系统内部索引三层资产
4. 保持检索方式可配置，但尽量减少用户频繁切换
5. 最大程度复用现有能力，优先改产品心智和资源分层

### 非目标

1. 第一版不做自动监听外部文件变化
2. 第一版不做多人协作和云同步
3. 第一版不让 AI 自动回写用户原始文件
4. 第一版不要求完全移除现有项目/知识模块内部实现，只要求用户侧心智完成切换

## 核心产品契约

### 1. 项目仍然存在，但只是 Vault 的入口壳

项目能力保留，用来承载：

- 创建项目
- 选择本地文件夹
- 保存项目级配置
- 打开上次绑定的 Vault

但项目不再是知识的主心智。知识的真相源是用户选中的本地文件夹。

### 2. 一个 Vault = 一个本地文件夹

用户可以：

- 选择一个已有文件夹
- 或让 GoodNight 创建一个新文件夹

一旦绑定后，GoodNight 围绕这个文件夹工作。

### 3. 主界面默认是原始文件树

第一版默认体验尽量贴近 Obsidian：

- 左侧是 Vault 的原始文件树
- 用户看到的就是实际文件和目录
- 不默认用“知识条目视图”替代文件树

### 4. 用户原始文件只读

AI 不直接改写用户原始文件。

当 AI 需要沉淀内容时：

- 只能创建新文件
- 不能覆盖或重写用户已有文件
- 后续如需编辑，必须由用户自己处理或另行确认

### 5. AI 产出与用户原始文件分层

用户可见的 AI 产出放在 Vault 可见目录中，且按 skill 分开。

系统内部索引和缓存放在隐藏目录中，用户默认不需要感知。

### 6. 手动刷新，而非自动监听

第一版不自动检测外部文件改动。

用户通过显式按钮手动刷新：

- 刷新文件树
- 刷新基础索引
- 视情况刷新当前 skill 索引

## 目录结构

假设用户选择的 Vault 根目录为 `<vault>/`，则 GoodNight 维护如下结构：

```text
<vault>/
  用户原始文件...

  _goodnight/
    outputs/
      llmwiki/
      rag/
      m-flow/

  .goodnight/
    config.json
    base-index/
    skills/
      llmwiki/
      rag/
      m-flow/
```

### `_goodnight/outputs/`

这是用户可见的 AI 产出目录。

按 skill 隔离：

- `_goodnight/outputs/llmwiki/`
- `_goodnight/outputs/rag/`
- `_goodnight/outputs/m-flow/`

建议所有由 AI 新建的用户可读文档都放在这里，例如：

- 问题总结
- 结构化知识页
- 需求文档
- 功能文档
- 流程文档

### `.goodnight/`

这是系统内部目录，默认隐藏，不在文件树中展示。

#### `.goodnight/config.json`

保存项目级知识库配置，例如：

- 默认检索方式
- 启用的 skills
- 输出规则
- 刷新策略
- 最近一次刷新时间

#### `.goodnight/base-index/`

保存所有 skill 共用的最小公共索引，例如：

- 文件指纹
- 路径映射
- 基础分块
- 时间戳
- 基础主题

这个层是公共底座，任何 skill 都依赖它。

#### `.goodnight/skills/<skill>/`

保存 skill 专属的内部索引和缓存，例如：

- `llmwiki` 的结构化知识页中间产物
- `rag` 的检索索引
- `m-flow` 的流程/相关性路径化索引

## Skill 体系

### 总体规则

GoodNight 不再只有一种“知识整理方式”，而是支持多个技能共用一个 Vault。

第一版支持：

- `M-Flow`
- `LLMWiki`
- `RAG`

### Skill 适配原则

第一版实现必须遵守以下原则：

1. 最大程度参考原始 skill 的能力边界、输入输出结构、使用方式和用户心智
2. 不为了统一而把所有 skill 硬压成同一种行为
3. 适配工作主要发生在 GoodNight 的 Vault、索引、输出目录和 UI 入口层
4. 复用的是 skill 的工作模型，而不是只借一个名字
5. 如原始 skill 与 GoodNight 的本地 Vault 模型冲突，则优先保留原 skill 的核心能力语义，再做最小必要适配

这意味着：

- `M-Flow` 要尽量保留其“相关性优先、路径化理解”的工作方式
- `LLMWiki` 要尽量保留其“结构化沉淀、长期知识页”的工作方式
- `RAG` 要尽量保留其“分块、召回、证据回答”的工作方式

### 默认 skill

项目默认检索方式设为 `M-Flow`。

原因：

- 用户不需要频繁切换
- M-Flow 更适合作为默认的“找相关信息、理解上下文、回答问题”能力
- 结构化沉淀和 wiki 风格整理交给 `LLMWiki`
- 纯召回检索交给 `RAG`

### Skill 参考来源

#### M-Flow

参考：

- `FlowElement-ai/m_flow`

适配要求：

- 相关性优先而非仅相似度
- 路径化检索思路
- 面向复杂上下文的 evidence routing
- 在 GoodNight 中作为默认检索方式
- 输出和缓存接入 GoodNight 的 Vault 分层，而不是改写其核心判断逻辑

#### LLMWiki

参考：

- `Astro-Han/karpathy-llm-wiki`

定位：

- 参考 Karpathy 的 LLM Wiki pattern
- 用于结构化知识沉淀
- 用于把原始资料整理成便于 AI 消费和用户阅读的新文档
- 尽量保留原始 LLM Wiki 的组织方式和产物风格
- 在 GoodNight 中改造成面向本地 Vault 和 `_goodnight/outputs/llmwiki/` 的实现

#### RAG

RAG 作为独立 skill 类型保留，不绑定单一唯一仓库。

适配要求：

- chunk + retrieval 的配置化能力
- 可替换的召回实现
- 面向问答的基础检索链
- 在 GoodNight 中作为可切换检索方式存在
- 与 `base-index` 共存，而不是重复发明底层文件扫描体系

## 检索方式配置

### 项目级默认配置

每个项目保存一个默认 skill。

第一版默认值：

- `m-flow`

### 运行时临时切换

用户在执行具体任务时可以临时切换 skill，但不鼓励频繁切换。

切换分成两个动作：

- `仅本次使用`
- `设为项目默认`

### 入口位置

检索方式入口放在文件树上方。

展示形式建议为：

- `检索方式：M-Flow`
- 右侧一个轻量 `更换` 入口

不做大面积 tab，不强化频繁切换。

### 切换提示

切换时提示：

> 切换检索方式会影响 AI 如何读取这个知识库。通常不需要频繁切换。若当前方式没有覆盖你的问题类型，再临时切换。

### 系统何时建议切换

系统默认不主动鼓励切换，只有以下情况才提示：

1. 当前 skill 索引还未构建
2. 当前问题类型明显更适合另一种 skill
3. 当前召回结果太弱，尝试另一种方式可能更好

## 交互流程

### 1. 创建项目

用户创建项目时执行：

1. 选择已有本地文件夹，或新建一个文件夹
2. GoodNight 绑定该文件夹为 Vault
3. 初始化：
   - `.goodnight/`
   - `_goodnight/outputs/`
4. 生成默认配置：
   - 默认 skill = `m-flow`

### 2. 首次进入项目

首次进入时主界面默认展示：

- Vault 原始文件树
- 文件树上方的检索方式栏位
- AI 面板

`.goodnight/` 默认隐藏，不显示在树中。

### 3. 手动刷新

用户点击 `刷新` 时执行：

1. 重新扫描 Vault 文件树
2. 重建或更新 `base-index`
3. 如当前默认 skill 或当前任务 skill 需要，再更新对应 skill 索引

### 4. 问答

默认流程：

1. 读取当前使用 skill
2. 优先读 `.goodnight/base-index/`
3. 再读取 `.goodnight/skills/<skill>/`
4. 必要时回源读取原始文件
5. 输出答案

如果索引较旧：

- 不阻断问答
- 但提示结果可能基于旧索引

### 5. 知识整理和文档生成

当用户要求整理或生成文档时：

1. 使用当前 skill 决定检索和组织方式
2. 绝不改写用户原始文件
3. 只在 `_goodnight/outputs/<skill>/` 中新建文档

## 各 Skill 的职责

### M-Flow

默认职责：

- 复杂问答
- 相关性检索
- 跨文件上下文理解
- 流程与路径分析

### LLMWiki

默认职责：

- 结构化整理
- 主题归纳
- 生成 wiki 风格的新知识文档
- 帮助形成长期沉淀材料

### RAG

默认职责：

- 常规检索问答
- 基础召回
- 面向文本块的快速定位

## 与现有实现的关系

### 保留

优先保留现有可复用能力：

- knowledge store / knowledge search 基础能力
- AIChat 路由能力
- 现有系统索引基础模块
- 现有项目壳与持久化能力

### 调整

需要重点调整：

1. 用户侧心智从“项目中的知识库页面”转为“本地 Vault 文件树”
2. 把现有系统索引放到 `.goodnight/base-index/` 语义下
3. 把 skill 能力改为真正可配置、可切换的 retrieval mode
4. 让 AI 新文档统一落在 `_goodnight/outputs/<skill>/`
5. skill 实现以“最大程度参考原方案，再适配 GoodNight”为基本约束

### 兼容策略

第一阶段允许内部继续借用现有项目与知识模块，只要用户看到的心智已经是：

- 打开文件夹
- 看文件树
- 用 skill 检索和生成

## 成功标准

以下条件成立时，说明设计方向正确：

1. 用户把 GoodNight 理解成“打开一个本地知识库文件夹”
2. 用户在主界面看到的是原始文件树，而不是抽象知识条目列表
3. 用户原始文件不会被 AI 直接改写
4. AI 产出与用户原始文件清晰分层
5. 默认检索方式是 M-Flow，且用户不需要频繁切换
6. 切换检索方式的入口足够明显，但不打扰日常操作
7. `.goodnight/` 对系统有用，但对用户几乎隐形

## 第一阶段实现范围

第一阶段只做足以完成心智切换的能力：

1. 项目创建/打开改为绑定本地文件夹
2. 主界面默认切为原始文件树
3. 文件树上方加入检索方式入口
4. 默认检索方式为 M-Flow
5. `.goodnight/` 与 `_goodnight/outputs/` 目录初始化
6. 用户原文件只读，AI 只创建新文件
7. 手动刷新文件树和索引

不在第一阶段内的内容：

1. 自动监听外部变更
2. 云同步
3. 多人协作
4. 高级 skill 市场或第三方安装机制
