# AI 设置保存入口与模型切换设计

## 摘要

本次设计只解决两个明确的交互问题：

- AI 设置页同时出现“保存”和“保存设置”，造成重复和困惑
- 同一个供应商下虽然有多个模型候选，但保存与切换路径不够直观

目标是把 AI 设置页收敛成“编辑当前配置草稿，然后显式点击一次保存”的模式，并确保单个供应商下可以方便地切换模型并保存当前选择。

## 目标

- 设置页只保留一个显式保存入口，文案统一为 `保存`
- 保留“测试连接”和“启用 / 关闭”动作，但它们不再和第二个保存入口并列制造歧义
- 在单个供应商配置下支持多模型切换
- 模型切换既支持点击候选项，也支持手动输入模型 ID
- 保存后当前配置的供应商、Key、Base URL、模型、请求头一起落盘

## 非目标

- 不引入“一个配置同时绑定多个模型轮询调用”的能力
- 不改造全局配置结构，不新增供应商分组数据模型
- 不调整聊天区“当前 AI”选择器的基本含义，它仍然选择一条已启用配置
- 不处理本次需求之外的设置页视觉重构

## 当前问题

### 保存入口重复

当前设置详情头部存在一个“保存设置”按钮，底部动作区又存在一个“保存”按钮。两者都执行保存语义，但界面没有表达它们在职责上的差异，用户会自然理解为重复按钮。

### 供应商与模型关系表达不够直接

当前设置页已经可以为一个配置维护模型字段，也能展示模型候选，但交互上没有把“这是同一供应商下的多个模型切换”表达得足够清楚。用户容易把配置切换、供应商切换、模型切换看成三套平级动作。

## 产品原则

- 一个配置是一条可保存、可启用的 AI 接入记录
- 一个配置在任意时刻只对应一个当前模型
- 供应商决定模型候选来源，模型决定该配置当前实际调用的模型 ID
- 用户编辑字段时先停留在草稿态，只有点击 `保存` 才写回配置
- `启用 / 关闭` 是状态动作，不替代保存
- `测试连接` 是校验动作，不替代保存

## 交互设计

### 保存动作

设置详情区只保留一个主保存按钮，文案统一为 `保存`。该按钮放在底部动作区，与 `启用 / 关闭`、`测试连接` 并列。

详情头部不再出现第二个保存按钮，也不再保留“保存设置”文案。

点击 `保存` 时：

- 将当前草稿中的名称、provider、API Key、Base URL、model、custom headers 写回当前配置
- 同步当前 provider 下的模型候选缓存，便于后续继续切换
- 给出“已保存当前配置”的反馈

### 供应商与模型

用户在设置页中选择某个配置后：

- 可以修改该配置的 provider 类型与 Base URL
- 可以拉取该 provider 的模型列表，或使用预置模型候选
- 可以点击模型 chip 切换当前模型
- 也可以直接手动输入模型 ID

这些操作都只更新当前草稿，直到用户点击 `保存` 才持久化。

### 启用与关闭

`启用 / 关闭` 继续保留，但职责明确为“让当前配置出现在聊天区可选列表中”。

- 如果草稿尚未保存，点击 `启用` 前先沿用当前逻辑，将必要字段写回配置
- 如果配置不完整，则不给启用，并提示先补全 API Key 与模型

这保证现有行为兼容，但主路径仍然是“先保存，再启用”。

## 系统设计

### 数据模型

沿用现有 `AIConfigEntry`：

- 不新增“一个配置多个已保存模型”的字段
- `model` 仍然是当前配置的单值字段

模型候选继续使用按 `provider + baseURL` 建立的运行时缓存。这样可以支持：

- 同一供应商的多模型候选展示
- 切换后重新进入设置页仍能看到已拉取过的候选

### 状态流

1. 用户在设置页选择某条配置
2. 组件根据该配置生成 `settingsDraft`
3. 用户修改 provider、Base URL、model 等字段
4. 模型候选区根据当前 provider 与 baseURL 组合展示对应模型列表
5. 用户点击 `保存`
6. store 调用 `updateConfig` 持久化当前草稿
7. 如果该配置已启用且被聊天区选中，运行时配置随之更新

## 实现边界

本次实现应保持改动最小：

- 删除设置详情头部的“保存设置”按钮
- 底部仅保留一个 `保存` 按钮作为保存入口
- 保持现有 `handleApplySettings` 作为唯一保存实现
- 保持现有模型候选、拉取模型、手输模型能力
- 只在必要处补测试，验证界面源码不再出现双保存入口，并保留模型切换相关实现

## 测试

至少验证以下场景：

- 设置详情区源码中不再同时出现头部保存按钮和底部保存按钮
- 仍保留唯一保存动作 `handleApplySettings`
- 模型输入框仍绑定 `settingsDraft.model`
- 模型候选点击后仍会把所选模型写入 `settingsDraft.model`
- 拉取模型后仍会更新当前 provider 对应的模型候选缓存

## 已确认决策

- 保存交互采用单一显式保存按钮，而不是自动保存
- 同一供应商下允许切换多个模型，但一次只保存一个当前模型
- 聊天区继续按“配置”切换，不改成按“供应商”切换
