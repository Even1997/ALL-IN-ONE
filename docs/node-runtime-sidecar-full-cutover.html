ALL-IN-ONE / Full Cutover Proposal

Node Runtime Sidecar
一次性整包迁移方案

目标不是继续把 React 做得更聪明，而是把 runtime 从前端主线程里完整搬出去。最终形态是 React 只订阅和渲染，Node.js sidecar 负责 turn orchestration、tool loop、MCP、审批、回放和持久化，Tauri 只保留桌面壳和 sidecar 生命周期。

纯 Node.js sidecar UI subscribes runtime HTTP + WebSocket protocol Tauri shell only
方案定位

不是渐进式发布路线，而是单次 cutover 版本。迁移分支内部允许兼容层，正式发布时旧 runtime 路径必须整体删除。

参考来源

结构参考了 cc-haha-main 的本地服务端 + WebSocket 事件流思路，但运行时选择更稳的 Node.js，不做 Bun-only 依赖。

最终目标

把当前“组件里驱动 runtime”的模型替换成“协议驱动 runtime”，让性能问题和架构问题一起消失，而不是继续在前端补丁式优化。

为什么这次要整包改

Current

运行时边界在前端

6k+

AIChat.tsx 级别的大组件同时承担 UI、streaming、审批、回放、tool execution glue 和 store 写入，边界天然会发散。

Current

状态真源混在 UI 层

4+

会话、runtime、审批、MCP 由多个前端 store 拼投影，任何局部事件都容易引发大范围重组和重渲染。

Decision

不要再修局部热区

1

这次直接把运行时搬到 Node.js sidecar。这样性能收益来自职责重划，不再依赖组件层面 endless memo。

目标架构

Layer 1

Desktop UI

  • 只发 command，只收 event，只渲染 projection
  • 保留 input draft、panel state、selection 这类纯交互态
  • 不再执行 tool loop、runtime replay、approval orchestration
Layer 2

Shared Runtime Protocol

  • 统一 command / event / snapshot schema
  • 本地 HTTP + WebSocket，sidecar 绑定 127.0.0.1 随机端口
  • Tauri 负责注入 auth token 和 sidecar 连接信息
Layer 3

Node.js Sidecar Runtime

  • session lifecycle、turn coordinator、tool execution、MCP、approval、replay、memory
  • 把现有 TS runtime 逻辑整体迁走，收口成服务端执行模型
  • SQLite 作为唯一持久化底座
Layer 4

Tauri Shell

  • 负责 sidecar 生命周期、路径、窗口、通知、更新、原生能力
  • 不再持有 runtime 业务 store 和业务命令

事件流

当前和目标的差异

Current

前端是 runtime 宿主

  • AIChat.tsx 和前端 runtime 模块共同驱动 turn lifecycle
  • 多个 Zustand store 既保存 UI 态也保存 runtime 真状态
  • Tauri Rust 还承担 runtime JSON store 和业务命令
  • 任何 streaming、审批、状态 tick 都容易影响 UI 主线程
Target

前端是 runtime 订阅者

  • UI 只通过协议发 command，收 event 和 snapshot
  • Node sidecar 是唯一执行和持久化真源
  • Tauri 退回为桌面壳与 sidecar 生命周期管理层
  • 性能优化来自边界重划，不再依赖组件级补丁

一次性切换怎么做

Rule
只发布一次新架构

迁移分支内允许短期兼容层，但正式 cutover 版本不保留旧 runtime 路径，不做双系统长期共存。

1. 先搭新骨架

切成 workspace，建立 apps/runtime、packages/runtime-protocol、packages/runtime-client、packages/runtime-core。

2. 再搬执行内核

把现有 TS runtime orchestration、tool loop、context、memory、MCP、execution graph 整体迁入 Node sidecar。

3. 最后切桌面前端

桌面 UI 改成只连 sidecar 协议；同时删除 Rust runtime store、删除前端旧执行入口、跑一次 legacy 数据迁移。

大风险和对应控制

Risk 1

Sidecar 打包与启动失败

  • Tauri 启动前做 health check
  • sidecar 版本和桌面版本强绑定
  • 保持本地日志目录和 crash dump
Risk 2

旧数据迁移不完整

  • 写一次性 migration runner
  • 先迁 session，再迁 replay/approval/checkpoint
  • 迁移后生成校验摘要
Risk 3

协议漂移

  • command/event 都有 schema 和版本号
  • 桌面端只依赖 protocol package
  • 不允许 UI 偷偷读 runtime 内部结构
Risk 4

切完仍然卡

  • UI 只持 projection cache
  • 消息流通过事件局部更新
  • 运行态 tick 不再驱动大组件重算

结论

如果目标是“来就全部做完”，最优方案不是继续修补当前 React runtime，而是一次性切成 Desktop UI / Shared Protocol / Node.js Sidecar Runtime / Tauri Shell 四层。这样你拿到的不只是更顺滑的 UI，而是一套后面还能继续扩展本地 agent、远程驱动、headless automation 和多终端接入的真正底座。
